# 团队架构与技能编写方法论

这是产品专家 Agent 的**内部方法论库**，回答两个问题：

1. **怎么把一个复杂任务拆成协作的子代理团队**（团队架构模式）
2. **怎么写出/测试一个高质量的技能**（技能编写与测试）——尤其用于 `/经验写回`、`/更新请求` 自我进化时生成新技能

它不是面向用户的产品能力，而是支撑现有五项能力（调研/策划/AI策划/Demo/SQL）的**编排与自我进化方法论**。

> **来源与许可**：本方法论蒸馏并适配自 [revfactory/harness](https://github.com/revfactory/harness)（Apache-2.0，作者 @revfactory）。harness 原插件是 Claude Code 专用的（依赖 `TeamCreate`/`SendMessage`/`TaskCreate` 等 agent-team 原语）；本仓库是 **Cursor 原生**，因此这里把"Agent Teams"运行时映射为我们的 **并行子代理批次 + 主代理编排**（与 `skills/research-toolkit/protocols/subagent-search.md` 一致），方法论内核保持不变。原始证据：结构化预配置在 15 个软件任务上的作者自测 A/B 显示平均质量 +60%（49.5 → 79.3，n=15，作者自测，第三方复现待验证）。

## 第一部分：团队架构模式

### 0. 运行时映射（Claude Code → Cursor）

| harness 概念 | 我们的实现 |
|---|---|
| Agent Teams（`TeamCreate` + `SendMessage` 互相通信） | 主代理用 `Task` 工具按批次并行派发子代理；子代理把结果汇总回主代理，由主代理转达与裁决 |
| 共享任务列表（`TaskCreate`/`TaskUpdate`） | 主代理维护 TODO + 任务目录里的过程文档（如 `05-synthesis/SUBAGENT_ROSTER.md`） |
| 子代理（`Agent` 直接调用） | 同样是 `Task` 工具，单发、只回主代理 |

实操约束（沿用 `subagent-search.md`）：**单批最多 4 个子代理并行**；子代理之间不直接通信，由主代理做信息中转与冲突裁决；每个子代理只负责一个明确数据面/职责，必须落标准产物。

### 1. 模式选择决策树

```
任务需要 ≥ 2 个角色协作吗？
├── 否（单角色）        → 直接做，或单个子代理
└── 是 → 角色之间是否强顺序依赖？
        ├── 是          → 流水线 Pipeline
        └── 否 → 是否同一输入、多视角并行？
                ├── 是  → 扇出/扇入 Fan-out/Fan-in
                └── 否 → 是否需要按输入类型路由到不同专家？
                        ├── 是 → 专家池 Expert Pool
                        └── 否 → 是否"先产出后检验"？
                                ├── 是 → 生成-检验 Producer-Reviewer
                                └── 否 → 运行时动态分配？→ 监督者 Supervisor / 层级委托 Hierarchical
```

### 2. 六种模式

| 模式 | 结构 | 适合 | 本仓库对应 |
|------|------|------|-----------|
| **Pipeline 流水线** | 分析→设计→实现→检验，前一步产出是后一步输入 | 强顺序依赖 | PRD 章节流水线（合同→撰写→配图→校验）；调研 Phase 0-7 |
| **Fan-out/Fan-in 扇出扇入** | 同输入并行多视角 → 汇总 | 多平台/多角度并行 | 调研 Phase 2 子代理并行采集（国内+海外通道）→ 汇总 summary |
| **Expert Pool 专家池** | 路由器按类型选专家 | 输入类型决定处理方式 | 调研 Phase 0 自动识别任务类型 → 路由到对应 task-type |
| **Producer-Reviewer 生成-检验** | 生成→检验→（有问题）重生成 | 质量需独立把关 | 采集门禁/成稿门禁的独立 `research-review` 子代理；三轮事实核查 |
| **Supervisor 监督者** | 中央代理看进度动态派活 | 工作量可变、运行时决定分配 | 主代理按采集进度补派子代理 |
| **Hierarchical 层级委托** | 总控→组长→实务，递归分解 | 问题天然分层 | 复杂全链路任务（调研→策划→Demo）的分层推进 |

要点：
- **Fan-out/Fan-in 是并行的首选**，汇总环节的质量决定整体质量。
- **Producer-Reviewer 必须设最大重试次数（2~3 次）**，防止无限循环——与门禁的"最多 5 轮"一致。
- **层级委托深度 ≤ 2 层**，超过 3 层会丢上下文。
- 实战常是**复合模式**：如"扇出 + 生成-检验"（并行采集 → 各自独立审核）、"流水线 + 扇出"（顺序阶段里某段并行）。

### 3. 子代理 vs 主代理直办

| 情形 | 选择 |
|------|------|
| 需要多视角交叉、实时修正调查方向 | 并行子代理批次（首选） |
| 一次性、完全隔离、无需互通的取数 | 单个子代理 |
| 1-2 步能搞定、无编排收益 | 主代理直接做，不要为拆而拆 |

### 4. 子代理拆分/复用判据

| 判据 | 拆分 | 合并 |
|------|------|------|
| 专业性 | 领域不同则拆 | 领域重叠则合 |
| 并行性 | 可独立执行则拆 | 顺序强依赖则合 |
| 上下文 | 上下文负担大则拆 | 轻快则合 |
| 复用性 | 多处复用则拆 | 仅此一处考虑合 |

新建子代理职责前，先查是否与现有职责重复；一个子代理聚焦一个职责，复用性最高、重复最少。

## 第二部分：技能编写方法论

用于 `/经验写回`、`/更新请求` 沉淀新技能，或新增/重构现有 skill 时。

### 1. Description 是唯一触发器

Agent 只看 `name + description` 决定要不要用这个技能。原则：

- 写清**技能做什么** + **具体触发场景**，并标明**边界条件**（什么相似情形不该触发）
- 略带"主动性"措辞，补偿模型偏保守的触发判断
- 复杂、多步、专业的任务更容易触发；过于简单的任务即使 description 完美也可能不触发

反例：「处理数据的技能」（太模糊）。正例：列出具体动作 + 触发措辞 + 边界。

### 2. 本文写作风格

- **Why-First**：先讲为什么，模型理解原理后在边界情况也能判断对。不要只堆 `ALWAYS/NEVER` 规则。
- **一般化**：从反馈/测试发现问题时，在**原理层**修，不要只补丁式适配某个测试样例（避免过拟合）。
- **命令式语气**：用"……做""……禁止"，技能是指示书。
- **节约上下文**：每句都要问"模型是否已知？没有它会出错吗？一个例子是否胜过长篇解释？"——能删则删。

### 3. Progressive Disclosure（渐进式披露）

技能体量大时，把细节下沉到 `references/`，主 `SKILL.md` 只留概览 + 路由：

- **按域分文件**：用户问财务就只加载 `finance.md`
- **条件式加载**：「需要追踪变更时 → 参见 references/redlining.md」
- **大文件加目录**：≥ 300 行的 reference 顶部放目录

（本仓库 research-toolkit 的 `protocols/` + `references/` 就是这个模式。）

### 4. 脚本打包判据

观察测试中子代理的轨迹，出现以下信号就把它固化进 `scripts/`：

| 信号 | 处理 |
|------|------|
| 多个测试都生成同一个辅助脚本 | 打包进 `scripts/` |
| 每次都跑同样的依赖安装 | 技能里写明依赖安装步骤 |
| 反复同样的多步做法 | 写成技能正文的标准流程 |
| 每次同类报错后用同样的绕法 | 写进"已知问题与解决法" |

### 5. 不要放进技能

README/CHANGELOG 等附属文档、技能生成过程的元信息（测试结果/迭代历史）、面向用户的说明书、模型已知的通用常识。

## 第三部分：技能测试与迭代

核心循环：**写 → 跑测试 → 评估 → 改进 → 重测**。

### 1. With-skill vs Baseline 对照

对每个测试 prompt 同时跑两路：带技能 vs 不带技能（或改前的快照版本），对比产出质量与 token/耗时。这是 harness「+60%」证据的来源方法。

### 2. 测试 prompt 写法

要像**真实用户会输入的、具体自然的句子**，覆盖：核心用例 1 个 + 边界用例 1 个 +（可选）复合任务 1 个。混合正式/口语、显式/隐式意图、简单/复杂。

### 3. Assertion 评分

- 好 assertion：客观可判真假、描述性命名、验证技能的**差异化价值**
- 警惕 **non-discriminating assertion**（带不带技能都 100% 通过的，如"产出存在"）——测不出技能价值，要替换成更有挑战的
- 可编程验证的就写成脚本，比肉眼快且可复用

### 4. Description 触发验证

写 ~20 条触发 eval：should-trigger 10 条 + should-NOT-trigger 10 条，重点放**边界/近似**用例（关键词相似但应由别的技能处理）。并检查新技能 description 是否与现有技能触发区重叠，重叠就把边界条件写得更清楚。

> 本仓库已有现成的测试基线：仓库级 `tests/test_product_expert_agent.py` + 各 skill 的 `tests/test_*.py`。新技能写回时，应配套契约测试（存在性 + 关键标记 + 触发词），并跑通全量测试后再合并——这就是我们落地的"技能测试"。

## 第四部分：运维增量（借鉴 ECC）

以下思路借鉴自 [affaan-m/ECC](https://github.com/affaan-m/ECC)（MIT，跨 harness 操作层）。我们只取**方法论**，不引入其框架代码——ECC 是整套操作系统，与本仓库"轻量聚焦"定位不同，整体内置会冲突。

### 1. 持续学习（Instincts）增强 `/经验写回`

把单次会话里反复奏效的做法，提炼成带**置信度**的"本能（instinct）"，攒够证据再固化为技能：

- 触发信号：同一类做法在多次任务里重复出现、或同一类错误反复用同一绕法解决
- 沉淀格式：`动作（做什么）+ 证据（哪几次任务验证过）+ 适用场景 + 置信度`
- 升级路径：本能（低成本记录）→ 多次验证、置信度够高 → 经 `/经验写回` 升级为正式技能（配套契约测试）
- 与第三部分衔接：升级为技能时走"With-skill vs Baseline"验证

### 2. Token 优化

- 模型分级路由：简单转换/确定性任务用轻量模型或直接编辑，复杂推理/架构才上重模型
- 系统提示瘦身 + 上下文是公共资源：每句都问"模型已知吗？没有它会出错吗？"（与第二部分"节约上下文"一致）
- 长任务用后台/子代理隔离上下文，主代理只收摘要

### 3. 并行（worktree / cascade）

- 多个**互不依赖**的改动，用 git worktree 在隔离工作树并行推进，最后合流
- 与第一部分"扇出/扇入"配合：并行产出 → 汇总环节统一收口
- 何时扩实例：任务可清晰切成独立块、且块数 ≥ 3 时收益明显；1-2 块不值得拆

### 4. 验证循环（evals）

- **检查点式**（checkpoint）：阶段结束跑一次评估闸（我们的采集门禁/成稿门禁就是）
- **连续式**（continuous）：每步小评估，适合高风险/不可逆操作
- pass@k：同一任务跑多次看通过率，识别高方差（不稳定）流程
- 落到本仓库：写回技能必须配契约测试 + 跑通全量，就是最小可执行的 eval 闸

### 5. 安全自检

涉及配置/指令/脚本写回时，按 `agent-security-scan.md` 跑 `/安全扫描`（AgentShield 连接器），把安全检查纳入自我进化闭环。

## 何时查阅本方法论

- 一个任务需要**多子代理协作/并行**时 → 查第一部分选模式（并行参见第四部分）
- 执行 `/经验写回`、`/更新请求` **生成或重构技能**时 → 查第二、三部分 + 第四部分持续学习
- 评审一个写回 PR 是否合格时 → 对照第三部分的测试要求 + 第四部分安全自检（见 `submission-review-contract.md`）
