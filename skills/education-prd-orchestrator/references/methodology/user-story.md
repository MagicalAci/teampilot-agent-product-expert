# 用户故事方法论

## 什么是用户故事

用户故事是从用户视角表达需求的最小单元，格式固定为：

> **作为** [角色]，**我想要** [行为/能力]，**以便** [价值/目的]

用户故事不是功能描述，不是技术需求，不是解决方案。它描述的是"用户想完成什么"，而不是"系统要做什么"。

## 何时使用

- 用户画像完成后，将画像的目标和痛点转化为可执行的需求
- 从抽象需求到具体功能的桥梁
- 团队需要就需求范围达成一致
- 作为功能列表和优先级排序的输入

## INVEST 原则

好的用户故事满足 INVEST 标准：

| 原则 | 含义 | 检查方式 |
|------|------|---------|
| **I**ndependent | 故事之间尽量独立 | 能否单独交付、单独验收 |
| **N**egotiable | 细节可协商 | 不是固定合同，可以讨论实现方式 |
| **V**aluable | 对用户有价值 | 能说清"用户为什么在乎" |
| **E**stimable | 可估算工作量 | 团队能给出粗略评估 |
| **S**mall | 足够小 | 一个迭代内可完成 |
| **T**estable | 可测试 | 有明确的验收标准 |

## 用户故事编写步骤

### Step 1：从画像出发

对每个用户画像，列出其核心目标和痛点，每个目标/痛点转化为 1-3 个用户故事。

### Step 2：写故事主体

标准格式：

```
作为 [画像名/角色]，
我想要 [具体行为]，
以便 [达成的价值]。
```

错误示范：
- ❌ "作为用户，我想要一个按钮" → 这是功能描述
- ❌ "作为用户，我想要系统用 React 开发" → 这是技术决策
- ❌ "作为产品经理，我想要提升留存" → 这是业务指标

正确示范：
- ✅ "作为新用户，我想要在 3 步内完成首次体验，以便快速判断产品是否适合我"
- ✅ "作为付费用户，我想要看到我的使用数据和进步趋势，以便感到投入是值得的"

### Step 3：补充验收标准

每个故事至少附 2-5 条验收标准（Acceptance Criteria），使用 Given-When-Then 格式：

```
Given [前置条件]
When [用户操作]
Then [预期结果]
```

### Step 4：按主题归组

将故事按用户旅程阶段或功能域归组，形成 **Epic → Story** 结构：

- Epic 是一组相关故事的集合
- 一个 Epic 通常对应一个完整的用户目标

## 故事地图

用户故事地图（Story Mapping）是按两个维度组织故事的方法：

- **横轴**：用户旅程的时间顺序
- **纵轴**：每个阶段从核心到增强的优先级

```
时间 →  注册     首次体验    日常使用    付费转化
────────────────────────────────────────────────
核心     注册流程   核心功能    常用功能    基础付费
增强     引导教程   个性化设置   高级功能    会员权益
锦上添花  社交绑定   成就系统    社区互动    推荐奖励
```

横向的第一行构成 MVP 的 **骨架**。

## 常见错误

- 故事只写"我想要"，不写"以便"
- 故事粒度太粗（一个故事包含 5 个功能）或太细（一个按钮一个故事）
- 用解决方案替代用户需求
- 没有验收标准，无法判断完成
- 故事之间有隐藏的强依赖
- 只写了 happy path，缺少异常场景

## 产出物

- 按 Epic 归组的用户故事列表
- 每个故事附验收标准
- 可选：用户故事地图
- 可选：故事间依赖关系标注
