# 用户画像方法论

## 什么是用户画像

用户画像是基于真实数据和研究构建的典型用户原型，用来代替模糊的"目标用户"概念，让团队对齐同一个具体的人。

画像不是人口统计学表格，而是对"这个人面对什么处境、想达成什么目标、用什么方式做决策"的结构化回答。

## 何时使用

- 产品定义前，需要明确"为谁设计"
- 团队对目标用户有分歧
- 需要从多类用户中聚焦优先服务对象
- 作为用户故事和用户旅程的前置输入

## 画像构建步骤

### Step 1：收集原始输入

可用来源：

- 用户访谈记录 / 客服对话
- 行为数据 / 埋点日志
- 问卷调查
- 竞品评论与口碑
- 销售 / 运营反馈
- 现有产品使用数据

没有一手数据时，标记为"假设型画像"，后续必须验证。

### Step 2：识别行为模式

从原始数据中提炼：

- 用户使用频率和场景
- 决策路径和关键触发点
- 信息获取渠道
- 对产品的期望与不满
- 付费意愿和决策因素

### Step 3：聚类与命名

将相似行为模式聚成 2-4 个画像组：

- 每个画像给一个记忆性名称（如"效率型妈妈""深度学习者"）
- 标注主画像（产品优先服务）和次画像
- 画像之间的差异必须体现在行为和需求上，不只是年龄性别

### Step 4：填充画像卡片

每个画像至少包含：

| 维度 | 必须回答的问题 |
|------|---------------|
| 基本属性 | 年龄段、职业、生活状态、技术熟练度 |
| 核心目标 | 使用这个产品想达成什么 |
| 当前做法 | 现在怎么解决这个问题 |
| 核心痛点 | 现有方案哪里不满意 |
| 决策因素 | 影响选择的关键考量 |
| 使用场景 | 在什么时间、什么地点、什么状态下使用 |
| 一句话概括 | 用一句话描述这个人的核心诉求 |

### Step 5：验证与迭代

- 拿画像回到真实用户数据验证是否匹配
- 请业务一线团队确认画像是否贴合
- 每个版本周期回顾画像是否需要更新

## 常见错误

- 画像只有人口统计，没有行为和目标
- 画像太多，失去聚焦作用
- 画像没有数据支撑，全靠想象
- 画像做完就归档，不参与后续决策
- 把"我们想要的用户"当成"真实用户"

## JTBD 视角补充

Jobs-to-be-Done 框架关注用户在特定情境下想完成的"任务"：

- **功能性任务**：需要完成什么具体事情
- **情感性任务**：希望自己感觉如何
- **社会性任务**：希望别人怎么看自己

JTBD 格式：`当 [情境] 时，我想要 [动机]，这样我就能 [期望结果]`

画像和 JTBD 不冲突：画像回答"谁"，JTBD 回答"为什么"。

## 产出物

- 2-4 张用户画像卡片
- 主画像标注
- 画像间差异对比表
- 数据来源和置信度标注
