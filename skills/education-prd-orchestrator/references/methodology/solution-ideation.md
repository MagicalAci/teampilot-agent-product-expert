# 方案构思方法论

## 什么是方案构思

方案构思是在核心痛点明确后，从多个视角生成可能的解决方案，然后收敛出最有前景的方案方向。它发生在"问题已经清楚"和"功能列表已经确定"之间。

## 何时使用

- 核心痛点已经抽象和排序完成
- 需要从问题空间转入解决方案空间
- 团队需要对齐"用什么方式解决"而不只是"解决什么"
- 作为核心假设和功能列表的前置输入

## 构思步骤

### Step 1：锁定问题域

从核心痛点抽象的产出中，选择本版本聚焦的 1-2 个问题域。

每个问题域要回答：

- 为谁解决（画像）
- 在哪个场景解决（旅程阶段）
- 解决到什么程度算成功

### Step 2：发散——多视角生成方案

从至少 3 个视角产出候选方案：

**产品视角**：
- 能否通过改善现有功能解决
- 能否通过新增功能解决
- 能否通过流程重新编排解决
- 能否通过信息架构调整解决

**用户视角**：
- 用户期望的最自然的交互方式是什么
- 竞品是怎么处理的，有什么可学可避
- 用户自己发明了什么"偏方"来绕过问题

**技术视角**：
- 当前技术栈能支持什么
- AI/自动化能在哪里介入
- 有没有低成本的快速验证方案

每个方案用一句话描述核心思路，附简要说明。

### Step 3：初筛——可行性过滤

快速过滤明显不可行的方案：

| 过滤条件 | 说明 |
|---------|------|
| 技术可行 | 当前能力和时间窗口内能做到 |
| 业务合规 | 不违反政策、法规或品牌约束 |
| 资源匹配 | 当前团队规模和技能覆盖 |
| 独立可交付 | 不依赖不可控的外部条件 |

### Step 4：收敛——方案对比

对通过初筛的 2-4 个方案进行结构化比较：

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 核心思路 | | | |
| 解决程度 | 彻底/部分/最小 | | |
| 用户感知 | 高/中/低 | | |
| 开发成本 | 大/中/小 | | |
| 风险 | 主要风险 | | |
| 依赖 | 需要什么前提 | | |
| 验证难度 | 容易/困难 | | |

### Step 5：选择方案方向

最终选择不一定是"最好的"，而是"当前条件下最值得尝试的"。

选择要回答：

- 为什么是这个方案而不是其他
- 这个方案验证失败后的 fallback 是什么
- 这个方案的最小可验证版本是什么

## How Might We（我们可以怎样）

发散阶段可以使用 HMW 问题格式来打开思路：

```
我们可以怎样 [让用户在 X 场景下更容易 Y]？
```

HMW 规则：
- 不能太宽（"怎样让用户满意"）
- 不能太窄（"怎样加一个按钮"）
- 不能包含解决方案（"怎样用 AI 来..."）
- 应该聚焦在用户目标上

## Crazy 8s 快速构思

限时 8 分钟，画出 8 个方案草图，每个 1 分钟：

- 不追求完美，追求数量
- 鼓励离谱的方案
- 完成后投票选出最有潜力的 2-3 个

## 常见错误

- 直接跳到第一个想到的方案
- 只从技术视角出发
- 发散不够就收敛
- 所有方案都是同一个方向的微调
- 选择方案时不考虑验证成本
- 方案描述过于模糊，无法评估

## 产出物

- 候选方案列表（每个一句话 + 简要说明）
- 方案对比矩阵
- 最终方案方向选择及理由
- 方案的最小可验证版本描述
