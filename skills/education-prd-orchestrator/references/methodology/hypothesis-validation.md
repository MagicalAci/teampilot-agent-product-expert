# 核心假设方法论

## 什么是核心假设

核心假设是方案能成立的前提条件。每个产品方案都隐含着一组"如果 X 成立，那么 Y 就能实现"的假设，但团队往往没有显式识别它们。

核心假设的作用是在投入大量开发资源之前，先明确"什么必须为真，这个方案才能成功"，然后优先验证风险最高的假设。

## 何时使用

- 方案构思完成，选定了方案方向
- 需要判断方案中哪些部分风险最高
- 在写 PRD 之前做一次风险预检
- 作为 V1 范围界定和功能优先级的决策依据

## 假设识别步骤

### Step 1：从方案中提取假设

对选定的方案，从四个维度提取隐含假设：

**价值假设（Value）**：
- 用户真的有这个问题吗？
- 用户愿意用我们的方式解决吗？
- 解决后用户感知到的价值足够吗？

**可用性假设（Usability）**：
- 用户能理解这个功能/交互吗？
- 用户能顺利完成关键流程吗？
- 学习成本在可接受范围内吗？

**可行性假设（Feasibility）**：
- 技术上能实现吗？
- 在预算和时间内能交付吗？
- 依赖的外部能力/数据是否可靠？

**商业可行性假设（Viability）**：
- 这个方案符合商业模式吗？
- 获客成本和 LTV 的关系能否成立？
- 是否与现有业务冲突？

### Step 2：假设结构化

每个假设用标准格式描述：

```
假设编号：H-001
假设描述：[明确、可验证的陈述]
假设类型：价值 / 可用性 / 可行性 / 商业可行性
当前置信度：高 / 中 / 低
如果不成立的影响：致命 / 需调整方案 / 影响局部
验证方式：[具体如何验证]
验证成本：高 / 中 / 低
```

### Step 3：优先级排序

使用 **风险-影响矩阵** 排序：

```
影响大 │  优先验证     │  优先验证
       │ （高风险高影响） │ （低风险高影响，观察即可）
───────┼──────────────┼──────────────
影响小 │  可以延后      │  可以忽略
       │ （高风险低影响） │ （低风险低影响）
       └──────────────┴──────────────
              高风险          低风险
```

优先验证象限：**高风险 + 高影响** 的假设必须在 PRD 定稿前完成验证。

### Step 4：设计验证方式

常用验证方法按成本排列：

| 方法 | 成本 | 适用场景 | 周期 |
|------|------|---------|------|
| 桌面研究 | 极低 | 市场数据、竞品证据 | 1-2 天 |
| 用户访谈 | 低 | 价值和需求验证 | 3-5 天 |
| 原型测试 | 中 | 可用性和交互验证 | 1-2 周 |
| A/B 测试 | 中高 | 已有产品的假设验证 | 2-4 周 |
| MVP / 灰度 | 高 | 完整方案验证 | 4-8 周 |
| 烟雾测试 | 低 | 用户兴趣和付费意愿 | 数天 |

### Step 5：记录验证结果

每个已验证假设更新：

- 验证结果：成立 / 不成立 / 部分成立
- 关键证据
- 对方案的影响：继续 / 调整 / 放弃
- 新增的后续假设

## 常见错误

- 团队认为"假设就是已知的事实"
- 只识别技术假设，忽略价值和可用性假设
- 所有假设都标"高优先级"，没有排序
- 验证方法和假设不匹配（用 A/B 测试验证"用户是否有这个问题"）
- 假设验证结果不影响后续决策
- 跳过假设验证直接写 PRD

## 产出物

- 假设清单（结构化）
- 风险-影响矩阵
- 验证计划（方法、负责人、时间）
- 验证结果与决策记录
