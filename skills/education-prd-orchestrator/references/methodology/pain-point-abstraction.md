# 核心痛点抽象方法论

## 什么是核心痛点抽象

核心痛点抽象是从分散的用户反馈、旅程图痛点、访谈记录中提炼出高层级、可设计的问题命题。它的目标是从"50 个零散不满"收敛为"3-5 个值得解决的核心问题"。

## 何时使用

- 用户画像和用户旅程完成后
- 有大量零散的痛点、反馈、需求待收敛
- 需要回答"这个版本到底要解决什么问题"
- 作为方案构思阶段的直接输入

## 抽象步骤

### Step 1：汇总原始痛点

把以下来源的痛点全部列出：

- 用户画像中的痛点
- 用户旅程图中标注的痛点
- 客服记录中的高频投诉
- 用户访谈中的关键抱怨
- 竞品评论中反复出现的问题
- 产品数据中反映的异常

每条痛点记录：

| 原始痛点 | 来源 | 涉及角色 | 涉及旅程阶段 | 频次/严重度 |
|---------|------|---------|------------|-----------|

### Step 2：去噪与合并

- 去除重复表述
- 合并同义不同表达的痛点
- 去除解决方案伪装成痛点的条目（"没有暗黑模式"是方案，"晚上看屏幕刺眼"是痛点）
- 去除产品无法影响的外部问题

### Step 3：向上归因

对每个痛点追问"为什么这是个问题"，找到更高一层的问题命题：

```
表层痛点：注册流程太长
  → 为什么是问题？新用户没有耐心
  → 更深层：用户在感知到价值之前就被要求付出成本
  → 核心命题：价值感知前置不足
```

追问 2-3 层即可，不要过度抽象到失去指导意义。

### Step 4：聚类与命名

将归因后的痛点聚成 3-5 个核心问题域：

每个问题域需要：

| 要素 | 说明 |
|------|------|
| 命名 | 一个简洁的问题标签 |
| 一句话定义 | 这个问题域在说什么 |
| 包含的原始痛点 | 哪些具体痛点归到这里 |
| 影响范围 | 影响哪些画像、哪些旅程阶段 |
| 严重度判断 | 是致命问题还是体验优化 |
| 当前状态 | 现在有没有解决方案、效果如何 |

### Step 5：排序与聚焦

排序依据：

- **影响广度**：影响多少用户、多少场景
- **影响深度**：问题不解决会导致什么后果（流失/投诉/替代）
- **可解决性**：当前团队是否有能力解决
- **战略对齐**：是否与产品方向一致

输出一个 2×2 矩阵或排序表，明确"这个版本优先解决哪 1-2 个核心问题域"。

## 痛点 vs 需求 vs 功能

| 层级 | 示例 | 谁定义 |
|------|------|--------|
| 痛点（Problem） | "不知道孩子学到了什么" | 来自用户 |
| 需求（Need） | "需要可视化的学习进度报告" | 产品团队抽象 |
| 功能（Feature） | "学习周报 + 知识点掌握度图表" | 产品+设计+技术定义 |

核心痛点抽象工作在 **痛点 → 需求** 这一层，不要过早跳到功能。

## 常见错误

- 把功能诉求直接当痛点（"需要导出 PDF"不是痛点）
- 只看高频，不看严重度
- 所有痛点平等对待，不做排序
- 归因过度，抽象到"用户不满意"这种无用结论
- 只看单角色，忽略多角色之间的冲突痛点

## 产出物

- 原始痛点汇总表
- 3-5 个核心问题域定义
- 问题域排序与本版本聚焦决策
- 每个问题域到原始痛点的追溯关系
