# 编排规则

## 主原则

这个 skill 只有一个主编排器。

主编排器负责：

- 决定当前阶段
- 读取对应 agent 规则
- 安排脚本做机械处理
- 安排用户检查点
- 统一收口版本与输出

主编排器不应该一上来把所有阶段一次性做完。

## 阶段顺序

默认严格按这个顺序推进：

1. 任务卡
2. 证据包
3. 用户画像
4. 用户故事
5. 用户旅程
6. 核心痛点抽象
7. 方案构思
8. 核心假设
9. 功能列表与优先级
10. 定义闸门
11. PRD 章节写作
12. 配图
13. 同步与校验

只有在上游稳定时，才允许跳到下游。

## 方法论知识库使用规则

`references/methodology/` 下的方法论文件是知识库，不是强制流程：

- 每个阶段开始时，按需读取对应的方法论文件
- 不要一次性预加载所有方法论
- 方法论提供思路和框架，具体执行根据项目实际调整
- 如果用户已经有现成的画像/故事/旅程等产物，优先复用而不是重做

## 什么时候切换到对应 Agent

- 要判断材料是否够：读 `Evidence Agent`
- 要构建用户画像：读 `Persona Agent`
- 要写用户故事：读 `Story Agent`
- 要画用户旅程：读 `Journey Agent`
- 要收敛痛点：读 `Pain Point Agent`
- 要发散方案：读 `Ideation Agent`
- 要识别假设：读 `Hypothesis Agent`
- 要排功能优先级：读 `Priority Agent`
- 要冻结产品定义：读 `Definition Agent`
- 要写 PRD 正文：读对应 `Chapter Agent`
- 要补图：读 `Diagram Agent`
- 要更新版本与索引：读 `Sync Agent`

## 上下游依赖规则

- 不要在没有画像的情况下写用户故事
- 不要在没有旅程和痛点的情况下构思方案
- 不要在方案未定时写功能优先级
- 不要在功能优先级未定时写 V1 范围
- 不要在产品定义不稳定时写 PRD 章节
- 不要在功能关系未定时画功能架构图

但如果用户已有上游产物，可以跳过该阶段直接继承。

## 用户检查点规则

只要出现以下任一情况，就必须暂停并回问：

- 关键截图缺失
- 当前产品体验未发生
- 关键材料未提供
- 有 2 个以上合理定义
- 样本量或证据覆盖明显不足
- 核心问题域需要用户确认
- 方案方向需要用户确认
- V1 范围需要用户确认

## 脚本规则

脚本只处理这些事：

- 统计
- 索引
- 导出
- 校验

脚本不要替代这些事：

- 产品判断
- 章节结论
- 角色定义
- 战略取舍
- 用户画像构建
- 方案选择

## 输出规则

每个阶段结束时，至少产出这 4 项：

- 本轮结论
- 使用的证据
- 修改了什么文件
- 下一步建议

如果下一步需要用户参与，要明确说：

- 缺什么
- 为什么缺
- 用户该做什么
- 做完后继续什么
