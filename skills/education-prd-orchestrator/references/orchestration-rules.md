# 编排规则

## 主原则

这个 skill 只有一个主编排器。

主编排器负责：

- 决定当前阶段
- 读取对应 agent 规则
- 安排脚本做机械处理
- 安排用户检查点
- 统一收口版本与输出

主编排器不应该一上来把所有章节一次性写完。

## 阶段顺序

默认严格按这个顺序推进：

1. 任务卡
2. 证据包
3. 定义闸门
4. 章节写作
5. 配图
6. 同步与校验

只有在上游稳定时，才允许跳到下游。

## 什么时候切换到对应 Agent

- 要判断材料是否够写：读 `Evidence Agent`
- 要冻结产品是什么：读 `Definition Agent`
- 要写正文：读对应 `Chapter Agent`
- 要补图：读 `Diagram Agent`
- 要更新版本与索引：读 `Sync Agent`

## 章节顺序规则

- 不要先写第五章再猜第三章
- 不要在产品定义不稳定时写 V1 边界
- 不要在功能关系未定时画功能架构图

## 用户检查点规则

只要出现以下任一情况，就必须暂停并回问：

- 关键截图缺失
- 当前产品体验未发生
- 关键材料未提供
- 有 2 个以上合理定义
- 样本量或证据覆盖明显不足

## 脚本规则

脚本只处理这些事：

- 统计
- 索引
- 导出
- 校验

脚本不要替代这些事：

- 产品判断
- 章节结论
- 角色定义
- 战略取舍

## 输出规则

每个阶段结束时，至少产出这 4 项：

- 本轮结论
- 使用的证据
- 修改了什么文件
- 下一步建议

如果下一步需要用户参与，要明确说：

- 缺什么
- 为什么缺
- 用户该做什么
- 做完后继续什么
