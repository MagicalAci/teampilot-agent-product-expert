# Agent Map

## Evidence Agent

### 目标

先把"能证明什么、不能证明什么"整理清楚。

### 输入

- 当前 PRD
- 会议纪要
- 截图
- CSV
- XLSX
- 话术或知识库文档
- 搜索记录或证据日志

### 动作

1. 盘点证据源
2. 必要时运行统计脚本
3. 输出证据摘要和缺口清单
4. 判断是否能进入下一阶段

### 输出

- 证据包摘要
- 缺口清单
- 建议下一步

---

## Persona Agent

### 目标

基于证据构建结构化用户画像。

### 方法论

参考 [methodology/user-persona.md](methodology/user-persona.md)

### 输入

- 证据包（访谈记录、客服数据、行为数据、竞品评论）
- 用户已有的画像描述（如有）

### 动作

1. 从证据中提取行为模式
2. 聚类为 2-4 个画像
3. 标注主画像
4. 标注数据置信度

### 输出

- 用户画像卡片（使用 `assets/user-persona-template.md`）
- 画像对比表
- 数据来源与置信度

### 检查点

- 如果证据不足以区分画像，停下来让用户补充
- 如果画像纯靠假设，必须标记为"假设型画像"

---

## Story Agent

### 目标

将画像的目标和痛点转化为用户故事。

### 方法论

参考 [methodology/user-story.md](methodology/user-story.md)

### 输入

- 用户画像卡片
- 痛点列表

### 动作

1. 对每个画像，列出核心目标和痛点
2. 转化为 "作为...我想要...以便..." 格式
3. 补充验收标准（Given-When-Then）
4. 按 Epic 归组

### 输出

- 用户故事列表（使用 `assets/user-story-template.md`）
- Epic 归组结构
- 故事间依赖标注

---

## Journey Agent

### 目标

绘制主画像的端到端体验旅程。

### 方法论

参考 [methodology/user-journey.md](methodology/user-journey.md)

### 输入

- 主画像
- 用户故事
- 证据包中的体验记录或截图

### 动作

1. 划分旅程阶段
2. 填充每阶段的行为、接触点、想法、情绪、痛点
3. 标注关键时刻（Aha Moment、决策点、摩擦点）
4. 如多角色，画多条旅程线并标注交互节点

### 输出

- 用户旅程图（使用 `assets/user-journey-template.md`）
- 关键时刻标注
- 痛点与机会列表

---

## Pain Point Agent

### 目标

从分散痛点收敛为 3-5 个值得解决的核心问题域。

### 方法论

参考 [methodology/pain-point-abstraction.md](methodology/pain-point-abstraction.md)

### 输入

- 画像中的痛点
- 旅程图中的痛点
- 证据包中的用户反馈

### 动作

1. 汇总所有原始痛点
2. 去噪与合并
3. 向上归因 2-3 层
4. 聚类为 3-5 个问题域
5. 排序（影响广度、深度、可解决性、战略对齐）

### 输出

- 原始痛点汇总表
- 核心问题域定义
- 问题域排序与聚焦决策

### 检查点

- 聚焦的问题域必须由用户确认

---

## Ideation Agent

### 目标

从核心痛点出发，生成并收敛解决方案方向。

### 方法论

参考 [methodology/solution-ideation.md](methodology/solution-ideation.md)

### 输入

- 核心问题域（已由用户确认）
- 画像和旅程上下文

### 动作

1. 从产品、用户、技术三个视角发散候选方案
2. 初筛不可行方案
3. 结构化对比 2-4 个可行方案
4. 输出推荐方案及理由

### 输出

- 候选方案列表
- 方案对比矩阵
- 推荐方案与最小可验证版本

### 检查点

- 方案方向必须由用户确认后才能继续

---

## Hypothesis Agent

### 目标

识别方案中的隐含假设，排序并设计验证方式。

### 方法论

参考 [methodology/hypothesis-validation.md](methodology/hypothesis-validation.md)

### 输入

- 选定的方案方向
- 画像和痛点上下文

### 动作

1. 从价值、可用性、可行性、商业可行性四维提取假设
2. 结构化描述每个假设
3. 用风险-影响矩阵排序
4. 设计验证方式

### 输出

- 假设清单（使用 `assets/hypothesis-template.md`）
- 风险-影响矩阵
- 验证计划

### 检查点

- 高风险假设的验证结果需用户确认
- 验证失败的假设需用户决策是否调整方案

---

## Priority Agent

### 目标

将方案拆解为功能列表，排序并划定 V1 范围。

### 方法论

参考 [methodology/feature-prioritization.md](methodology/feature-prioritization.md)

### 输入

- 方案方向
- 用户故事
- 假设验证结果

### 动作

1. 功能分解与归组
2. MoSCoW 粗分
3. RICE 或 ICE 精排
4. 标注依赖关系
5. 划定 V1 线

### 输出

- 功能优先级表（使用 `assets/feature-priority-template.md`）
- V1 范围定义
- Won't-have 列表
- 成功标准

### 检查点

- V1 范围必须由用户确认

---

## Definition Agent

### 目标

综合上游全部产物，在正文写作前冻结产品定义。

### 输入

- 用户画像
- 用户故事
- 用户旅程
- 核心痛点
- 方案方向
- 假设验证结果
- 功能优先级与 V1 范围

### 必须回答

- 一句话定义
- 各端角色（如涉及多角色）
- 第一版主链路
- 什么不是这个产品

### 产出建议

- 一段总定义
- 一张判断表
- 1-3 条排除项

### 闸门规则

只要定义还有分叉，就不进入 PRD 章节写作。

## Chapter 5 Agent

### 目标

写"功能定义"——PRD 中最核心的章节，直接决定开发和设计能否基于文档开工。**这个章节的质量直接等于 PRD 的质量。**

### 上游输入

- 功能优先级详细列表
- 用户故事

### 输出格式（强制，不可降级）

功能需求详情必须使用 `| 模块 | 需求描述 | 原型图 |` 表格格式。需求描述列使用 `<br>` 换行 + 全角空格 `　` 缩进的嵌套编号结构。**不允许用段落式描述、不允许用简单列表替代、不允许省略任何结构段落。**

完整的格式规范、结构模板和标杆示例见 `assets/prd-template.md` §5.2。

### 逐模块执行流程（强制）

每个功能模块必须按以下步骤逐个完成，**不得跳步、不得批量简写**：

**Step 1 - 模块拆解**：从功能优先级列表中取出当前模块，明确模块编号（F-XX）和名称。

**Step 2 - 骨架填充**：按 `assets/prd-template.md` §5.2.3 的结构模板，逐段填写：
  - 入口
  - 页面组成（从上到下每个可见元素）
  - 每个交互元素（独立编号，写清触发 → 执行 → 成功/失败/取消）
  - 数据加载与刷新
  - 空状态与异常

**Step 3 - 粒度自检**：对照 `assets/prd-template.md` §5.2.5 的 12 项自检清单逐项打勾。不满足 8 项以上的，必须立即补全，不得进入下一个模块。

**Step 4 - 标杆比对**：将写好的模块与 `assets/prd-template.md` §5.2.4 的 F-A01 Feed 标杆示例做粒度比较。如果明显简于标杆，必须加深。

**Step 5 - 原型图标注**：原型图列填写 `待设计`（PRD 初稿阶段）。原型设计完成后，导出 PNG 存入 `images/` 目录，替换为图片引用。

### 写作原则（7 条铁律）

1. **按组件逐一拆解**：页面从上到下，每个可见元素独立编号，不得合并
2. **按状态穷举分支**：每个交互元素写清成功/失败/取消的处理路径
3. **按条件说明激活**：按钮何时可点、何时灰化、什么条件触发变化
4. **数据来源标清**：每个展示字段说明从哪里来（后端接口 / 本地缓存 / 第三方 SDK）
5. **错误提示写全**：具体英文文案，不写"显示错误"这种模糊表述
6. **空状态必写**：无数据时的插画 + 文案 + 引导按钮（含按钮文案和点击行为）
7. **纯功能逻辑**：不写色值、字号、圆角、像素间距等设计规格

### 粒度红线

- 每个功能模块的需求描述必须包含至少 4 个一级编号段落（入口 + 页面组成 + 至少 1 个交互元素 + 空状态与异常）
- 页面组成段落必须逐个元素列出，每个元素至少标注内容和数据来源
- 交互元素的分支描述必须到第三层缩进（如 3.3.1 → a/b/c）
- 单个模块的需求描述少于 10 个 `<br>` 换行，视为粒度不合格
- **如果一个功能模块的描述读完后，设计师还需要回来问"这个按钮点了以后怎么办"——说明粒度不够**

### 完成标准

- 所有 Must-have 功能模块已填写完毕，格式统一
- 每个模块通过 §5.2.5 粒度自检清单（至少 8/12 项）
- 每个模块的描述可直接交给设计师出设计稿、开发写代码，无需再问产品
- 需求描述中无色值、字号等设计规格
- 原型图列已正确标注（初稿为 `待设计`，设计完成后替换为 PNG 引用）

## Diagram Agent

### 目标

生成总览图和章节配图。

### 默认类型

- 流程总览图
- 功能架构图
- 用户旅程可视化图
- 必要的章节结构图

### 规则

- 先有稳定结论，再画图
- 一张图只回答一个核心问题
- SVG 为源文件，PNG 为文档引用文件
- 默认输出到独立交付目录的 `images/`，如有仓库内镜像路径，再额外同步

---

## Sync Agent

### 目标

收口版本、日志、索引和引用。

### 动作

1. 更新主文档版本号和变更记录
2. 更新独立交付目录下的 review / handoff 说明
3. 检查上游产物（画像、故事、旅程、假设、优先级）与 PRD 的一致性
4. 检查图片引用
5. 如果宿主工作区存在产品索引，再额外同步 `docs/workspaces/product/INDEX.md`
6. 输出完成说明

### 完成标准

- 上游产物、PRD、图片、review 交付说明四者一致
- 如果存在仓库内镜像路径，镜像内容与独立交付目录一致
