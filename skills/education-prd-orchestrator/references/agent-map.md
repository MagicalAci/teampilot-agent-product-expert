# Agent Map

## Evidence Agent

### 目标

先把"能证明什么、不能证明什么"整理清楚。

### 输入

- 当前 PRD
- 会议纪要
- 截图
- CSV
- XLSX
- 话术或知识库文档
- 搜索记录或证据日志

### 动作

1. 盘点证据源
2. 必要时运行统计脚本
3. 输出证据摘要和缺口清单
4. 判断是否能进入下一阶段

### 输出

- 证据包摘要
- 缺口清单
- 建议下一步

---

## Persona Agent

### 目标

基于证据构建结构化用户画像。

### 方法论

参考 [methodology/user-persona.md](methodology/user-persona.md)

### 输入

- 证据包（访谈记录、客服数据、行为数据、竞品评论）
- 用户已有的画像描述（如有）

### 动作

1. 从证据中提取行为模式
2. 聚类为 2-4 个画像
3. 标注主画像
4. 标注数据置信度

### 输出

- 用户画像卡片（使用 `assets/user-persona-template.md`）
- 画像对比表
- 数据来源与置信度

### 检查点

- 如果证据不足以区分画像，停下来让用户补充
- 如果画像纯靠假设，必须标记为"假设型画像"

---

## Story Agent

### 目标

将画像的目标和痛点转化为用户故事。

### 方法论

参考 [methodology/user-story.md](methodology/user-story.md)

### 输入

- 用户画像卡片
- 痛点列表

### 动作

1. 对每个画像，列出核心目标和痛点
2. 转化为 "作为...我想要...以便..." 格式
3. 补充验收标准（Given-When-Then）
4. 按 Epic 归组

### 输出

- 用户故事列表（使用 `assets/user-story-template.md`）
- Epic 归组结构
- 故事间依赖标注

---

## Journey Agent

### 目标

绘制主画像的端到端体验旅程。

### 方法论

参考 [methodology/user-journey.md](methodology/user-journey.md)

### 输入

- 主画像
- 用户故事
- 证据包中的体验记录或截图

### 动作

1. 划分旅程阶段
2. 填充每阶段的行为、接触点、想法、情绪、痛点
3. 标注关键时刻（Aha Moment、决策点、摩擦点）
4. 如多角色，画多条旅程线并标注交互节点

### 输出

- 用户旅程图（使用 `assets/user-journey-template.md`）
- 关键时刻标注
- 痛点与机会列表

---

## Pain Point Agent

### 目标

从分散痛点收敛为 3-5 个值得解决的核心问题域。

### 方法论

参考 [methodology/pain-point-abstraction.md](methodology/pain-point-abstraction.md)

### 输入

- 画像中的痛点
- 旅程图中的痛点
- 证据包中的用户反馈

### 动作

1. 汇总所有原始痛点
2. 去噪与合并
3. 向上归因 2-3 层
4. 聚类为 3-5 个问题域
5. 排序（影响广度、深度、可解决性、战略对齐）

### 输出

- 原始痛点汇总表
- 核心问题域定义
- 问题域排序与聚焦决策

### 检查点

- 聚焦的问题域必须由用户确认

---

## Ideation Agent

### 目标

从核心痛点出发，生成并收敛解决方案方向。

### 方法论

参考 [methodology/solution-ideation.md](methodology/solution-ideation.md)

### 输入

- 核心问题域（已由用户确认）
- 画像和旅程上下文

### 动作

1. 从产品、用户、技术三个视角发散候选方案
2. 初筛不可行方案
3. 结构化对比 2-4 个可行方案
4. 输出推荐方案及理由

### 输出

- 候选方案列表
- 方案对比矩阵
- 推荐方案与最小可验证版本

### 检查点

- 方案方向必须由用户确认后才能继续

---

## Hypothesis Agent

### 目标

识别方案中的隐含假设，排序并设计验证方式。

### 方法论

参考 [methodology/hypothesis-validation.md](methodology/hypothesis-validation.md)

### 输入

- 选定的方案方向
- 画像和痛点上下文

### 动作

1. 从价值、可用性、可行性、商业可行性四维提取假设
2. 结构化描述每个假设
3. 用风险-影响矩阵排序
4. 设计验证方式

### 输出

- 假设清单（使用 `assets/hypothesis-template.md`）
- 风险-影响矩阵
- 验证计划

### 检查点

- 高风险假设的验证结果需用户确认
- 验证失败的假设需用户决策是否调整方案

---

## Priority Agent

### 目标

将方案拆解为功能列表，排序并划定 V1 范围。

### 方法论

参考 [methodology/feature-prioritization.md](methodology/feature-prioritization.md)

### 输入

- 方案方向
- 用户故事
- 假设验证结果

### 动作

1. 功能分解与归组
2. MoSCoW 粗分
3. RICE 或 ICE 精排
4. 标注依赖关系
5. 划定 V1 线

### 输出

- 功能优先级表（使用 `assets/feature-priority-template.md`）
- V1 范围定义
- Won't-have 列表
- 成功标准

### 检查点

- V1 范围必须由用户确认

---

## Definition Agent

### 目标

综合上游全部产物，在正文写作前冻结产品定义。

### 输入

- 用户画像
- 用户故事
- 用户旅程
- 核心痛点
- 方案方向
- 假设验证结果
- 功能优先级与 V1 范围

### 必须回答

- 一句话定义
- 各端角色（如涉及多角色）
- 第一版主链路
- 什么不是这个产品

### 产出建议

- 一段总定义
- 一张判断表
- 1-3 条排除项

### 闸门规则

只要定义还有分叉，就不进入 PRD 章节写作。

## Chapter 1 Agent

### 目标

写"背景与问题"。

### 上游输入

- 证据包

### 重点

- 当前产品/业务现状
- 核心问题定义与证据支撑
- 为什么现在要解决

### 完成标准

- 现状说清楚，有真实材料支撑
- 问题定义明确，不是空泛描述

## Chapter 2 Agent

### 目标

写"目标用户与场景"。

### 上游输入

- 用户画像
- 用户旅程

### 重点

- 目标用户画像及特征
- 核心使用场景
- 用户旅程与关键痛点

### 完成标准

- 场景不是凭空假设，每个场景有画像和旅程支撑
- 用户真实问题与产品能力有明确对位

## Chapter 3 Agent

### 目标

写"产品定义与方案"。

### 上游输入

- 方案构思产出
- 核心假设
- 定义闸门产出

### 重点

- 产品定位（是什么、不是什么）
- 方案核心策略
- 核心假设及验证状态
- 如涉及多角色：角色关系与边界

### 完成标准

- 这一章能作为后面章节唯一上游口径

## Chapter 4 Agent

### 目标

写"范围与边界"。

### 上游输入

- 功能优先级（V1 线 + Won't-have）

### 重点

- 做什么
- 不做什么
- 条件性纳入什么
- 成功标准

### 完成标准

- 可以直接指导取舍
- 不是愿望清单

## Chapter 5 Agent

### 目标

写"功能定义"——PRD 中最核心的章节，直接决定开发和设计能否基于文档开工。

### 上游输入

- 功能优先级详细列表
- 用户故事

### 输出格式（强制）

功能需求详情必须使用 `| 模块 | 需求描述 | 原型图 |` 表格格式。

需求描述列使用 `<br>` 换行 + 全角空格 `　` 缩进的嵌套编号结构。

参考 `assets/prd-template.md` 第五章中的格式规范和示例。

### 写作原则

1. **按组件逐一拆解**：页面从上到下，每个可见元素独立编号
2. **按状态穷举分支**：每个交互元素写清成功/失败/取消的处理
3. **按条件说明激活**：按钮何时可点、何时灰化、什么条件触发变化
4. **数据来源标清**：每个展示字段说明从哪里来（后端接口 / 本地缓存 / 第三方 SDK）
5. **错误提示写全**：具体文案，不写"显示错误"这种模糊表述
6. **空状态必写**：无数据时的图标、文案、引导按钮
7. **纯功能逻辑**：不写色值、字号、圆角、像素间距等设计规格

### 粒度标准

每个功能模块表格行的需求描述，至少包含：

- 入口（从哪进入）
- 页面组成（从上到下每个元素）
- 交互规则（每个可操作元素的行为分支）
- 条件激活规则
- 异常/边界处理
- 空状态

如果一个功能点的需求描述少于 5 个编号段落，说明粒度不够。

### 完成标准

- 功能之间的关系清楚
- 每个功能点的描述可直接交给设计师出设计稿、开发写代码，无需再问产品
- 需求描述中无色值、字号等设计规格
- 格式统一使用表格 + 嵌套编号

## Chapter 6 Agent

### 目标

写"流程与交互"。

### 上游输入

- 用户旅程
- 功能交互设计

### 重点

- 主链路流程

### 完成标准

- 核心流程有清晰路径
- 异常处理不缺位

## Chapter 7 Agent

### 目标

写"依赖与风险"。

### 上游输入

- 假设验证结果
- 证据缺口

### 重点

- 上下游依赖
- 已知风险与应对
- 后续版本规划

### 完成标准

- 风险不是模糊担心，有具体影响和应对
- 遗留问题清晰交代

## Diagram Agent

### 目标

生成总览图和章节配图。

### 默认类型

- 流程总览图
- 功能架构图
- 用户旅程可视化图
- 必要的章节结构图

### 规则

- 先有稳定结论，再画图
- 一张图只回答一个核心问题
- SVG 为源文件，PNG 为文档引用文件
- 默认输出到独立交付目录的 `images/`，如有仓库内镜像路径，再额外同步

---

## Sync Agent

### 目标

收口版本、日志、索引和引用。

### 动作

1. 更新主文档版本号和变更记录
2. 更新独立交付目录下的 review / handoff 说明
3. 检查上游产物（画像、故事、旅程、假设、优先级）与 PRD 的一致性
4. 检查图片引用
5. 如果宿主工作区存在产品索引，再额外同步 `docs/workspaces/product/INDEX.md`
6. 输出完成说明

### 完成标准

- 上游产物、PRD、图片、review 交付说明四者一致
- 如果存在仓库内镜像路径，镜像内容与独立交付目录一致
