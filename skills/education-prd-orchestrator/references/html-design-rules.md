# HTML 页面稿规则

## 目标

HTML 页面稿不是为了“画一个好看的页面”，而是为了：

- 更清楚地表达某个页面 / 功能怎么承接用户
- 把核心判断、模块结构、交互和证据绑定到同一个交付件里
- 让 reviewer 能直接截图、讨论、批注和插入 PRD

## 最小结构

每个页面稿至少包含：

1. 页面名和适用角色
2. 这页要证明的价值
3. 核心模块拆分
4. 关键交互 / 异常处理
5. 截图或结构图插入位
6. reviewer 看完后该得出的结论

## 插图规则

- 有真实截图时，优先插真实截图
- 没有真实截图时，可以先用结构草图或占位块，但不能放失效图片链接
- 每个截图位都要写“它证明什么”
- 图和文要一一对应，不能出现页面稿里有模块、正文里没提到

## 交付规则

- 默认输出到 `outputs/<slug>/html/`
- 同时在 PRD 中说明这份 HTML 稿对应哪一章、哪一节
- 如果宿主工作区本来就有项目文档库，可以把最终页面稿镜像到仓库内，但独立交付目录仍然是主版本

## reviewer 检查点

- 这页服务的是谁，是否一眼看清
- 页面结构是否对应正文里的模块和判断
- 是否写明了关键交互和异常路径
- 截图 / 草图是否能证明正文里的判断
- 是否仍然有 `[待填]` 占位没有收口
