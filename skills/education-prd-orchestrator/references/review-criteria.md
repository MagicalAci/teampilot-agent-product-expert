# 审核标准

这份清单用于审核"别人使用 `/产品策划` 或 `/产品策划校验` 后提交的最终文档"，不是审核 skill 包本身。

## 一票否决项

- [ ] 文档对象明确，没有把不同产品、端口、角色或阶段混成一件事
- [ ] 核心结论已经写清，不是资料堆砌、会议纪要转录或过程记录
- [ ] 关键判断有真实证据支撑，没有明显编造截图、数据、流程或用户反馈
- [ ] 文档已经可评审，没有大段占位符、待补章节、待补截图或明显未完稿状态

## 1. 上游产物质量

- [ ] 用户画像有数据支撑，不是纯假设；主画像已标注
- [ ] 用户故事符合 INVEST 原则，有验收标准
- [ ] 用户旅程覆盖关键阶段，情绪和痛点有依据
- [ ] 核心痛点经过抽象和排序，不是原始反馈的罗列
- [ ] 方案经过多角度对比后选定，有选择理由
- [ ] 核心假设已识别，高风险假设有验证计划或验证结果
- [ ] 功能优先级有明确评分和 V1 线，Won't-have 已列出
- [ ] 上游产物之间口径一致（画像名称、痛点编号、功能 ID 等）

## 2. 定义与结论

- [ ] 一开头就能看清这份文档在回答什么产品决策问题
- [ ] 产品定位、目标用户、核心场景和要解决的问题定义清楚
- [ ] 范围取舍明确，知道这版要做什么、不做什么
- [ ] 现有能力复用与新增能力关系讲清楚
- [ ] 结论型表达成立，读完能直接得到判断

## 3. 结构完整性

- [ ] 章节完整，至少覆盖背景/问题、目标用户/场景、方案、范围、功能、流程、风险
- [ ] 总领层存在，能快速看懂整体产品结构和模块关系
- [ ] 章节之间衔接顺，不互相打架，不前后改口径
- [ ] 如果是校验稿，缺口、风险、修改建议和优先级分得清楚

## 4. 证据与真实性

- [ ] 关键结论对应到了真实输入材料
- [ ] 证据放在该出现的位置支撑正文
- [ ] 体验结论与截图、流程描述、产品实际状态一致
- [ ] 猜测与事实分得开；无法确认的部分被明确标成待确认项

## 5. 图文与配图

- [ ] 该有的流程图、架构图、表格、对比图已经补齐
- [ ] 图的作用明确，要么解释流程，要么解释结构，要么支撑结论
- [ ] 配图与正文位置匹配
- [ ] 图文口径一致，中文表达统一

## 6. 功能与落地性

- [ ] 核心功能拆解清楚，模块关系和先后顺序明确
- [ ] 描述足以支撑后续设计、开发或评审继续往下走
- [ ] 依赖项、约束条件、上下游关系有交代
- [ ] 成功标准明确，知道上线后怎么判断是否成立

## 6.1 功能需求详情格式（第五章专项 · 一票否决）

**以下任何一项不通过，整个第五章打回重写，不进入后续章节。**

### 格式合规（全部必须通过）

- [ ] 使用 `| 模块 | 需求描述 | 原型图 |` 三列表格格式，不得增减列
- [ ] 需求描述使用 `<br>` 换行 + 全角空格 `　` 缩进的嵌套编号结构
- [ ] 不存在段落式描述或简单无序列表替代编号结构的情况
- [ ] 每个功能模块独占一行，不存在多个模块合并为一行的情况

### 结构完整性（每个模块必须通过）

- [ ] 每个模块包含「入口」段落，写明从哪里进入、触发条件
- [ ] 每个模块包含「页面组成」段落，从上到下逐个列出所有可见元素
- [ ] 每个可见元素标注了数据来源（后端接口/本地缓存/第三方 SDK/用户输入）
- [ ] 每个可交互元素（按钮/输入框/开关/列表项/手势）有独立编号段落
- [ ] 每个交互元素写到「触发 → 执行 → 成功/失败/取消」三个分支
- [ ] 成功分支覆盖了所有条件分叉（不存在"成功后跳转"这种模糊描述）
- [ ] 包含「数据加载与刷新」段落（如涉及列表/Feed：首次加载量、分页规则、刷新行为、返回保持）
- [ ] 包含「空状态与异常」段落

### 粒度达标（每个模块必须通过）

- [ ] 错误提示写了具体英文文案（如 `Failed to connect Google. Please try again.`），不存在"显示错误提示"
- [ ] 空状态写了插画 + 文案 + 引导操作（含按钮文案和点击后行为）
- [ ] 条件激活规则写明了何时可点、何时灰化
- [ ] 性能敏感操作有量化指标（如加载时间上限）
- [ ] 与 `assets/prd-template.md` §5.2.4 的 F-A01 Feed 标杆示例相比，粒度不低于 70%

### 原型图列（评审前必须通过）

- [ ] 所有 Must-have 模块的原型图列已填写（PRD 初稿阶段填 `待设计`）
- [ ] 原型设计完成的模块已替换为 PNG 引用（`![xxx](images/F-XX-xxx.png)`）
- [ ] 不存在原型图列为空或填写"无"的模块
- [ ] 多状态页面有对应的多张 PNG（命名后缀区分状态）
- [ ] PNG 文件实际存在于 `images/` 目录中，路径有效

### 设计规格隔离

- [ ] 无色值（`#xxxxxx`、`蓝色 500`）
- [ ] 无字号（`14px`、`16sp`）
- [ ] 无圆角（`12px`、`rounded-lg`）
- [ ] 无像素间距（`padding: 16px`）
- [ ] 无字重（`font-weight: 600`）

### 自动打回规则

以下情况直接打回，不进入补充修改流程：

1. **格式违规**：未使用三列表格，或使用段落式描述替代嵌套编号
2. **结构缺失**：任何一个 Must-have 模块缺少「入口」或「页面组成」或「空状态与异常」段落
3. **粒度严重不足**：任何一个模块的需求描述少于 4 个一级编号段落，或少于 10 个 `<br>` 换行
4. **模糊交互**：存在 3 处以上"显示错误"/"跳转页面"/"弹出提示"等未写具体内容的模糊描述
5. **无空状态**：超过 2 个模块缺少空状态描述

## 7. 交付完整性

- [ ] 文档标题、版本号、日期、状态等头部信息完整
- [ ] 上游产物文件齐全（画像、故事、旅程、假设、优先级）
- [ ] 版本记录、review handoff 或相关归档动作已经同步
- [ ] 图片、图表、附件路径有效
- [ ] 命名、目录、引用关系清楚

## 审核结论怎么判

- [ ] 通过：一票否决项全部通过，主要章节、上游产物、证据、图文和版本同步无明显缺口
- [ ] 补充后通过：主判断正确，但缺少少量截图、配图、版本同步或局部论证
- [ ] 打回重做：产品定义错位、没有总领层、没有真实证据，或核心结论与材料对不上
