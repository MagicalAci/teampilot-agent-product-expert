# 审核标准

这份清单用于审核"别人使用 `/产品策划` 或 `/产品策划校验` 后提交的最终文档"，不是审核 skill 包本身。

## 一票否决项

- [ ] 文档对象明确，没有把不同产品、端口、角色或阶段混成一件事
- [ ] 核心结论已经写清，不是资料堆砌、会议纪要转录或过程记录
- [ ] 关键判断有真实证据支撑，没有明显编造截图、数据、流程或用户反馈
- [ ] 文档已经可评审，没有大段占位符、待补章节、待补截图或明显未完稿状态

## 1. 上游产物质量

- [ ] 用户画像有数据支撑，不是纯假设；主画像已标注
- [ ] 用户故事符合 INVEST 原则，有验收标准
- [ ] 用户旅程覆盖关键阶段，情绪和痛点有依据
- [ ] 核心痛点经过抽象和排序，不是原始反馈的罗列
- [ ] 方案经过多角度对比后选定，有选择理由
- [ ] 核心假设已识别，高风险假设有验证计划或验证结果
- [ ] 功能优先级有明确评分和 V1 线，Won't-have 已列出
- [ ] 上游产物之间口径一致（画像名称、痛点编号、功能 ID 等）

## 2. 定义与结论

- [ ] 一开头就能看清这份文档在回答什么产品决策问题
- [ ] 产品定位、目标用户、核心场景和要解决的问题定义清楚
- [ ] 范围取舍明确，知道这版要做什么、不做什么
- [ ] 现有能力复用与新增能力关系讲清楚
- [ ] 结论型表达成立，读完能直接得到判断

## 3. 结构完整性

- [ ] 章节完整，至少覆盖背景/问题、目标用户/场景、方案、范围、功能、流程、风险
- [ ] 总领层存在，能快速看懂整体产品结构和模块关系
- [ ] 章节之间衔接顺，不互相打架，不前后改口径
- [ ] 如果是校验稿，缺口、风险、修改建议和优先级分得清楚

## 4. 证据与真实性

- [ ] 关键结论对应到了真实输入材料
- [ ] 证据放在该出现的位置支撑正文
- [ ] 体验结论与截图、流程描述、产品实际状态一致
- [ ] 猜测与事实分得开；无法确认的部分被明确标成待确认项

## 5. 图文与配图

- [ ] 该有的流程图、架构图、表格、对比图已经补齐
- [ ] 图的作用明确，要么解释流程，要么解释结构，要么支撑结论
- [ ] 配图与正文位置匹配
- [ ] 图文口径一致，中文表达统一

## 6. 功能与落地性

- [ ] 核心功能拆解清楚，模块关系和先后顺序明确
- [ ] 描述足以支撑后续设计、开发或评审继续往下走
- [ ] 依赖项、约束条件、上下游关系有交代
- [ ] 成功标准明确，知道上线后怎么判断是否成立

## 6.1 功能需求详情格式（第五章专项）

- [ ] 使用 `| 模块 | 需求描述 | 原型图 |` 三列表格格式
- [ ] 需求描述使用 `<br>` 换行 + 全角空格缩进的嵌套编号结构
- [ ] 每个功能点覆盖：入口、页面组成、交互规则、结果分支、条件激活、异常处理
- [ ] 交互分支写到「点击 → 发生什么 → 成功/失败/取消各怎样」
- [ ] 错误提示写了具体文案（如 `Failed to connect Google. Please try again.`），不是模糊的"显示错误"
- [ ] 空状态写了图标 + 文案 + 引导操作
- [ ] 无色值（`#xxxxxx`）、无字号（`xxpx`）、无圆角、无像素间距等设计规格
- [ ] 数据字段标注了来源（后端/缓存/第三方 SDK）

## 7. 交付完整性

- [ ] 文档标题、版本号、日期、状态等头部信息完整
- [ ] 上游产物文件齐全（画像、故事、旅程、假设、优先级）
- [ ] 版本记录、review handoff 或相关归档动作已经同步
- [ ] 图片、图表、附件路径有效
- [ ] 命名、目录、引用关系清楚

## 审核结论怎么判

- [ ] 通过：一票否决项全部通过，主要章节、上游产物、证据、图文和版本同步无明显缺口
- [ ] 补充后通过：主判断正确，但缺少少量截图、配图、版本同步或局部论证
- [ ] 打回重做：产品定义错位、没有总领层、没有真实证据，或核心结论与材料对不上
