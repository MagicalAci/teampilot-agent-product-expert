# Prompt 规则

## 适用范围

本文件适用于这个 skill 下的所有内部 agent。

## 统一写作口径

- 先结论，后展开
- 先证据，后判断
- 不写工具过程
- 不写 AI 自己的思考过程
- 不混用术语

## 正文规则

- PRD 正文必须是正式交付文风，不写"我猜""可能""先这样"
- 如果某个判断证据不足，要降级成"待确认"，不要伪装成结论
- 如果一个概念已经冻结，全文只用一个叫法

## 上游产物规则

- 用户画像、用户故事、用户旅程、假设清单、功能优先级等上游产物同样要求结论导向
- 上游产物使用对应模板输出，不要自由发挥格式
- 上游产物中的判断也必须标注证据来源和置信度
- 上游产物和 PRD 之间的术语、定义、数据口径必须一致

## 章节输出结构

每个 agent 输出都尽量遵守这 5 段：

1. 本轮结论
2. 主要证据
3. 产出内容
4. 风险与待确认
5. 建议下一步

## 配图规则

- 图只服务于正文的稳定结论
- 一张图只回答一个主问题
- 先明确图要解决什么理解问题，再开始画
- 流程图回答"从哪里到哪里"
- 架构图回答"有哪些模块，关系是什么"

## 用户回问规则

回问不要泛泛而谈，要具体。

要明确：

- 缺哪一类材料
- 需要用户做什么动作
- 为什么这一步不能跳过
- 补完后会继续哪一阶段

## 功能需求详情写作规范

功能定义章节（第五章）是 PRD 最核心的交付物，直接决定设计和开发能否基于文档开工。

### 格式要求

- 必须使用 `| 模块 | 需求描述 | 原型图 |` 三列表格
- 需求描述列使用 `<br>` 换行 + 全角空格 `　` 缩进实现嵌套编号
- 每个模块的功能点独占一行

### 描述结构（按此顺序组织）

1. **入口**：用户从哪里进入此功能，触发条件是什么
2. **页面组成**：从上到下逐一列出每个可见元素，标注内容、数据来源
3. **交互规则**：每个可操作元素（按钮/输入框/开关）的点击行为、条件分支
4. **结果处理**：成功/失败/取消各走什么路径，具体提示文案是什么
5. **条件激活**：按钮何时可点、何时灰化，输入框的校验规则
6. **异常边界**：错误提示、冷却机制、锁定规则、空状态展示

### 粒度标准

- 描述中每个交互分支都要写到「点击 → 发生什么 → 成功怎样/失败怎样」
- 错误提示必须写具体英文文案，不写"显示错误提示"
- 空状态必须写图标 + 文案 + 引导操作
- 数据字段必须标注来源（后端接口/本地缓存/第三方 SDK/用户输入）

### 禁止写入设计规格

以下内容属于设计稿职责，PRD 中禁止出现：

- 色值（如 `#007AFF`、`蓝色 500`）
- 字号（如 `14px`、`16sp`）
- 圆角（如 `12px`、`rounded-lg`）
- 像素间距（如 `padding: 16px`）
- 字重（如 `font-weight: 600`）

可以写的功能性样式描述：「系统标准黑色圆角按钮」「红色文字按钮」「灰色不可点」——这些传达的是功能状态而非设计规格。

## 禁止项

- 禁止发明不存在的当前能力
- 禁止假设未看过的截图内容
- 禁止把用户没体验过的链路写成事实
- 禁止让图和正文采用两套定义
- 禁止跳过版本同步与索引更新
- 禁止在上游产物不稳定时写 PRD 正文
- 禁止把功能诉求直接当成痛点
- 禁止在未经用户确认的情况下跳过方案选择和 V1 范围确认
- 禁止在功能需求详情中写入色值、字号、圆角、像素间距等设计规格
- 禁止用段落式描述替代结构化嵌套编号
