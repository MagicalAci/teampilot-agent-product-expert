# Prompt 规则

## 适用范围

本文件适用于这个 skill 下的所有内部 agent。

## 统一写作口径

- 先结论，后展开
- 先证据，后判断
- 不写工具过程
- 不写 AI 自己的思考过程
- 不混用术语

## 正文规则

- PRD 正文必须是正式交付文风，不写"我猜""可能""先这样"
- 如果某个判断证据不足，要降级成"待确认"，不要伪装成结论
- 如果一个概念已经冻结，全文只用一个叫法

## 上游产物规则

- 用户画像、用户故事、用户旅程、假设清单、功能优先级等上游产物同样要求结论导向
- 上游产物使用对应模板输出，不要自由发挥格式
- 上游产物中的判断也必须标注证据来源和置信度
- 上游产物和 PRD 之间的术语、定义、数据口径必须一致

## 章节输出结构

每个 agent 输出都尽量遵守这 5 段：

1. 本轮结论
2. 主要证据
3. 产出内容
4. 风险与待确认
5. 建议下一步

## 配图规则

- 图只服务于正文的稳定结论
- 一张图只回答一个主问题
- 先明确图要解决什么理解问题，再开始画
- 流程图回答"从哪里到哪里"
- 架构图回答"有哪些模块，关系是什么"

## 用户回问规则

回问不要泛泛而谈，要具体。

要明确：

- 缺哪一类材料
- 需要用户做什么动作
- 为什么这一步不能跳过
- 补完后会继续哪一阶段

## 功能需求详情写作规范（第五章 · 强制）

功能定义章节（第五章）是 PRD 最核心的交付物，直接决定设计和开发能否基于文档开工。**以下所有规则均为强制执行，不可降级。**

### 格式要求（不可更改）

- 必须使用 `| 模块 | 需求描述 | 原型图 |` 三列表格，不得增减列
- 需求描述列使用 `<br>` 换行 + 全角空格 `　` 缩进实现嵌套编号
- 每个功能模块独占一行，不得将多个模块合并为一行
- 不得用段落式描述替代结构化嵌套编号
- 不得用简单无序列表（`- xxx`）替代编号结构

### 描述结构（按此顺序，每个段落必须存在）

每个模块的需求描述必须包含以下一级编号段落，缺少任何一个视为不合格：

1. **入口**：用户从哪里进入此功能，触发条件是什么
2. **页面组成（从上到下）**：逐个列出每个可见元素，每个元素标注：
   - 展示内容
   - 数据来源（后端接口 / 本地缓存 / 第三方 SDK / 用户输入）
   - 默认值 / 初始状态
   - 条件变化（什么条件下改变内容或状态）
3. **交互元素**（每个可交互元素独立编号）：每个元素写清：
   - 元素形态（按钮/输入框/开关/列表项/手势）
   - 触发动作（点击/长按/滑动/输入）
   - 执行流程 → 成功分支（含所有条件分叉）/ 失败分支（具体英文错误文案 + 重试逻辑）/ 取消分支
   - 条件激活（何时可点、何时灰化、冷却时间）
4. **数据加载与刷新**：首次加载量、分页/无限滚动规则、下拉刷新行为、返回保持滚动位置
5. **空状态与异常**：
   - 空状态：每种无数据场景的插画 + 文案 + 引导操作
   - 网络异常：离线/弱网的降级策略
   - 权限异常（如适用）：未登录/无权限的处理

### 粒度红线

- 每个模块至少 4 个一级编号段落
- 每个交互元素的分支描述至少到第三层缩进
- 错误提示必须写具体英文文案（如 `Failed to load. Tap to retry.`），禁止写"显示错误"
- 空状态必须写插画 + 文案 + 引导按钮（含按钮文案和点击后行为）
- 数据字段必须标注来源
- 性能敏感的操作必须写量化指标（如首屏加载时间上限）
- **终极检验标准：设计师读完后能直接画稿、开发读完后能直接写代码，无需返回问产品**

### 原型图工作流（强制）

原型图列在 PRD 生命周期中有明确的状态流转：

```
PRD 初稿 → 原型图列填 "待设计"
    ↓
设计师 / Demo 阶段完成原型设计
    ↓
从 Figma / 设计工具导出 PNG（不接受 JPG/SVG/链接）
    ↓
存入交付目录 images/ 下，命名：F-XX-模块名.png
    ↓
替换原型图列为 ![F-XX 模块名](images/F-XX-模块名.png)
    ↓
多状态页面：每个状态单独 PNG，后缀区分
  例：F-A01-feed-default.png, F-A01-feed-empty.png, F-A01-feed-offline.png
```

规则：
- PRD 评审前，所有 Must-have 模块的原型图列必须已替换为 PNG 引用
- 禁止原型图列长期留空或填写"无"
- 禁止用 Figma 链接替代 PNG 引用（链接可能失效，PRD 必须自包含）

### 禁止写入设计规格

以下内容属于设计稿职责，PRD 中禁止出现：

- 色值（如 `#007AFF`、`蓝色 500`）
- 字号（如 `14px`、`16sp`）
- 圆角（如 `12px`、`rounded-lg`）
- 像素间距（如 `padding: 16px`）
- 字重（如 `font-weight: 600`）

可以写的功能性样式描述：「系统标准黑色圆角按钮」「红色文字按钮」「灰色不可点」——传达功能状态而非设计规格。

### 逐模块写作节奏（强制）

写第五章时必须**一个模块写完、自检通过后再写下一个**，禁止：
- 先写完所有模块的骨架再回头补细节（会导致粒度不足）
- 同时写多个模块（会导致交叉引用混乱）
- 跳过自检直接进入下一个模块（会导致粒度逐步退化）

## 禁止项

- 禁止发明不存在的当前能力
- 禁止假设未看过的截图内容
- 禁止把用户没体验过的链路写成事实
- 禁止让图和正文采用两套定义
- 禁止跳过版本同步与索引更新
- 禁止在上游产物不稳定时写 PRD 正文
- 禁止把功能诉求直接当成痛点
- 禁止在未经用户确认的情况下跳过方案选择和 V1 范围确认
- 禁止在功能需求详情中写入色值、字号、圆角、像素间距等设计规格
- 禁止用段落式描述替代结构化嵌套编号
- 禁止跳过功能需求详情的逐模块自检流程（写完一个、检查一个、通过后再写下一个）
- 禁止批量写完所有模块骨架后再回头补细节
- 禁止原型图列长期留空或填写"无"
- 禁止用 Figma 链接替代 PNG 图片引用
- 禁止在错误提示处写"显示错误"等模糊表述，必须写具体英文文案
- 禁止省略空状态描述（每个模块必须有空状态 + 异常处理段落）
