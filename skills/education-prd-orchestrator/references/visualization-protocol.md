# 可视化协议

## 原则

- 可视化服务判断，不服务炫技
- 一张图只回答一个核心问题
- 有真实截图时，先下沉真实截图
- 结构图、流程图、图表都必须能回指到正文里的具体判断

## 什么时候用什么图

### 流程图

用于回答：

- 从哪里到哪里
- 谁先触发谁
- 一条主链路如何流动

默认输出到：`outputs/<slug>/images/`

### 功能架构图

用于回答：

- 当前做哪些功能
- 功能之间关系是什么
- 哪些是核心模块，哪些是支撑模块

默认输出到：`outputs/<slug>/images/`

### 数据图表

用于回答：

- 量化差异在哪里
- 哪组样本支持哪个判断
- 搜索 / 评论 / 证据汇总如何转成结构化论证

默认输出到：`outputs/<slug>/charts/`

## 图表入稿规则

- 图表必须写明数据来源、时间范围和样本量
- 只有图没有说明，不算完成
- 样本不足时，图标题和图注要显式写 `待确认`
- 图表颜色、命名、图注风格要统一

## 文件规则

- 结构图优先保留 SVG 源文件，再导出 PNG 供正文引用
- 图表 HTML、原始 CSV、导出 PNG / SVG 要能互相对应
- 正文只能引用已经存在的文件，不允许写未来路径

## reviewer 检查点

- 图类型是否匹配它要回答的问题
- 图注是否说明“这张图证明什么”
- 图片路径是否有效
- 图和正文是否使用同一套产品定义 / 指标定义
