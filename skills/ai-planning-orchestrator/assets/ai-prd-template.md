## 输入数据结构定义

> 本章要回答的问题：AI 最终接收什么输入、每个字段代表什么、哪些字段必填、哪些字段允许回退、哪些写法需要兼容。

### 1. 总数据结构

这一节必须放一份**完整、可运行、与真实脚本口径一致**的 JSON 示例。

写法要求：

- 只保留结构，不要塞演示性质的长说明
- 字段名必须与脚本真实读取字段完全一致
- 如果有兼容旧协议和新协议，两种都要在文中说明，但主示例只放一种"推荐标准"

```json
{
  "[rootFieldA]": {},
  "[rootFieldB]": []
}
```

### 2. 一级对象字段说明

对每个一级对象单独开小节，例如：

- `[rootFieldA]`
- `[rootFieldB][]`
- `[nestedFieldC]`

每个小节都使用同一张表：

| 字段 | 类型 | 是否必填 | 含义 | 来源 | 回退 / 备注 |
|---|---|---|---|---|---|
| `[fieldName]` | `string / number / boolean / array / object` | 是 / 否 | 这个字段表达什么业务事实 | 上游接口 / 埋点 / 脚本推导 | 缺失时如何处理 |

### 3. 数组项与嵌套对象字段

如果一级对象里还嵌套数组或子对象，继续拆开写，不要把所有字段堆在一张总表里。

建议按"对象粒度"拆节：

- `[arrayField][]`
- `[arrayField][].childObject`
- `[arrayField][].childArray[]`

### 4. 字段回退口径

这一节必须单独写，不能只散落在表格备注里。

建议用以下格式：

- `[fieldA]`
  - 优先取：`[task-level field]`
  - 缺失时回退：`[subject-level field]`
  - 再缺失时：`[兜底规则]`
- `[fieldB]`
  - 优先取：`[...]`
  - 缺失时回退：`[...]`

### 5. 输入红线

列出输入协议里**不能混淆**、**不能省略说明**、**不能写错归属**的点。

常见写法：

- 哪些字段只属于某个对象层级
- 哪些字段不能出现在另一侧模块
- 哪些图片 / 提示 / 标签只能透传，不能 AI 编造
- 哪些旧字段仍兼容，但新接入不推荐继续用

如果这是多阶段生成，必须显式写成顺序链，而不是一句"整体由 AI 生成"。

### 4. 排序公式与规则

推荐写法：


公式后必须接"这个公式实际在表达什么"，逐项解释，不要只给数学式。

## 提示词模块

> 本章要回答的问题：每个 AI 模块具体吃什么、吐什么、Prompt 怎么约束、输出如何映射到前端或下游。

### 1. 模块总表

必须使用以下列顺序：

| 模块 | 输入参数 | 输出参数 | 输出示例 | 前端映射 | prompt详情 |
|---|---|---|---|---|---|
| `[模块A]` | `[列字段]` | `[列字段]` | `[JSON 片段]` | `[前端字段 / API字段]` | `[prompt 文档路径 / 完整正文]` |

### 2. 每个模块怎么写

每个模块至少补齐 5 件事：

1. 输入参数：列清真实输入字段，不写模糊词
2. 输出参数：列清模块自己负责生成的字段
3. 输出示例：给一个最小 JSON 片段
4. 前端映射：说明最终落到哪个 UI / API 字段
5. prompt详情：可放链接，也可直接放正文，但全篇要统一

### 3. Prompt 正文标准结构

每个 Prompt 必须使用同一结构：

```md
## 人设
[这个 Agent 是谁，负责什么语气和职责]

## 任务
[它只负责生成哪些字段]

## 规则
[可执行的判断规则]

## 输出规范
[JSON 结构、长度、格式]

## 严格禁止
[不能编造什么、不能输出什么]

## 思考链
[内部步骤，只指导模型，不进入最终输出]
```

### 4. Prompt 写作硬要求

- `人设` 只定义角色与语气，不夹带输出格式
- `任务` 只定义模块产出，不写运行时外挂字段
- `规则` 必须可判定，避免"写得自然一点"这类模糊要求
- `输出规范` 必须写 JSON 结构、字段长度、返回边界
- `严格禁止` 单独列段，专门约束编造、越界、解释性文本
- `思考链` 只写内部处理步骤，不允许暴露给最终输出
