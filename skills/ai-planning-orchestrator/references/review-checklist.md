# 评审清单

## 命令定位

- [ ] `README` 和 `SKILL.md` 已明确四条主命令：`/AI策划`、`/AIPRD`、`/AI脚本`、`/AI测试`
- [ ] 全流程、PRD、脚本、测试四种模式的边界清楚，不互相混淆
- [ ] 旧命令如果保留，只作为兼容说明，不抢主入口

## AI PRD

- [ ] 顶部元信息完整：时间、平台、版本号、撰写人
- [ ] 章节顺序严格是 `输入数据结构定义 -> AI策略 -> 提示词模块 -> AI脚本`
- [ ] 输入结构部分写清总结构、一级对象、数组项、回退口径
- [ ] AI 策略写清生成顺序、平台配置、业务红线
- [ ] 模块策略写清候选池、排序公式、去重规则、字段回退口径
- [ ] 提示词模块表格包含 `模块 / 输入参数 / 输出参数 / 输出示例 / 前端映射 / prompt详情`
- [ ] Prompt 正文统一为 `人设 / 任务 / 规则 / 输出规范 / 严格禁止 / 思考链`
- [ ] AI脚本部分包含测试 JSON、运行入口、输出示例、压缩包说明

## AI脚本

- [ ] 已建立在冻结的字段 / Prompt 合同之上
- [ ] 至少包含核心脚本、样例、README / 调用文档
- [ ] 开发调用文档写清 CLI / HTTP / route 的适用入口
- [ ] 脚本包清单已列清哪些文件直接可用、哪些仍需目标项目补
- [ ] 没有在字段未冻结时直接落代码

## AI测试报告

- [ ] 明确区分单元测试、真实调用、兼容性验证
- [ ] 不只贴日志路径，而是有测试结论
- [ ] 写明当前模型是否支持 `json_schema` 或其他结构化输出方式
- [ ] 写明失败时是否有降级策略
- [ ] 写明剩余风险和限制

## 压缩包

- [ ] 至少包含生成器、客户端、handler 示例、测试、样例、Prompt、README / 调用文档
- [ ] README 说明"哪些可以直接用，哪些要目标项目自己补"
- [ ] 已提供脚本包清单或等价文件列表
- [ ] 平台配置与文档一致
- [ ] 输出示例与 PRD 字段一致

## 提示词策略卡（写提示词前）

- [ ] 策略已拆成带编号的可判定规则（区分硬/软约束）
- [ ] 每条规则一对一映射到提示词实现方式，无"隐含覆盖"
- [ ] 覆盖率 ≥95%，每条可回答"模型做到了吗？是/否"
- [ ] 冲突规则已声明优先级；六段 Prompt 位置布局正确（硬约束开头、禁止+格式末尾）

## 评测集

- [ ] 单一 scope（一个任务/route 一个集）
- [ ] 四桶齐备且配比接近 60/15/15/10（生产/对抗/边界/失败回放）
- [ ] 规模达 route 下限（≥30 起步），已脱敏、已去污染
- [ ] 每条至少 1 条可判定断言；事故用例进失败回放桶
- [ ] 跑过 `aggregate_eval_dataset.py` 体检无 schema 错误

## 评测体系与报告

- [ ] 维度+权重+阈值明确，每条可判真假
- [ ] grader 分层：规则能判的不上 judge（确定性地板）
- [ ] 用 judge：judge≠被测模型族、模型+量规版本钉死、有金标校准(κ 达标)
- [ ] 成对比较做位置交换；量规含反啰嗦判据
- [ ] 评测报告含分维度、pass@k、失败用例与聚类、与基线回归对比

## 调优报告

- [ ] 失败归因到根因(ASI)，不是只记"错了"
- [ ] 本轮变更 ≤ 3–5 处，回溯到策略卡规则（一般化修，非过拟合）
- [ ] 有 A/B 对比数据证明不退步；未回退已验证有效的改动
- [ ] 收敛标准达"良好"（覆盖率/合规/达标率/迭代轮次）

## 一致性

- [ ] 文档里的模型名、APPID、agent ID、chat_url 与代码一致
- [ ] 文档里的输出字段与示例 JSON 一致
- [ ] 文档里的验证结论与实际测试结果一致
- [ ] Prompt 文档、AI PRD、脚本输出没有互相打架
- [ ] 评测集断言、量规维度与 PRD/Prompt 口径一致
