# 评测驱动开发 SOP（EDD in AI策划）

> 把 `policies/llm-eval-methodology.md` 的方法论落到 `/评测集`、`/AI评测`、`/AI调优` 三条可执行流程。
>
> 何时读：用户要"建评测集 / 跑评测 / 出评测报告 / 调优"时；或 `/AI策划` 走到测试之后还要做质量度量与回归时。

---

## 命令与产物

| 命令 | 做什么 | 默认产物 |
|---|---|---|
| `/评测集 [对象]` | 汇总/构建/体检评测集 | `eval-dataset.jsonl` + 评测集体检报告 |
| `/AI评测 [对象]` | 搭评测体系 + 跑自动化评测 | 评分量规 + `eval-result.json` + 评测报告 |
| `/AI调优 [对象]` | 基于评测报告做定向调优 + 回归 | 新版提示词/服务 + 调优报告 |

三者与现有链路的关系：

```
/AIPRD → /AI脚本 → /AI测试（软件能跑）→ /评测集 → /AI评测（质量好不好）→ /AI调优（变更）→ /AI评测（回归）
```

**`/AI测试` 与 `/AI评测` 的区别**（不要混）：

| | /AI测试 | /AI评测 |
|---|---|---|
| 问的问题 | 脚本/接口能不能跑、输出是不是合法 JSON | LLM 输出**好不好**、能否回归、能否调优 |
| 手段 | 单测 + 真实调用 + 结构化兼容 | 评测集 + 量规 + grader + pass@k + A/B |
| 产物 | 测试报告 | 评分量规 + 评测报告 + 调优报告 |

## `/评测集` 工作流

目标：产出一个**单一 scope、四桶齐备、规模达线、已脱敏去污染**的 `eval-dataset.jsonl`。

1. **定 scope 与 route**：这套集只测一个任务；按意图/子任务切 route 分别建。
2. **四桶采集**（占比 60/15/15/10，见方法论 2.1）：
   - 生产样本：从真实流量分层采样，**脱敏**（去 PII）。
   - 对抗样本：越狱/注入/PII 提取等（可参考 OWASP LLM Top 10）。
   - 边界用例：领域专家手写长尾。
   - 失败回放：历史事故/线上 bug → 每个转 3–5 条复现用例。
3. **写断言**：每条至少写 1 条可判定断言（`contains`/`json_valid`/`json_schema_keys`/`regex`…）；有标准答案的填 `reference`。
4. **体检**：跑 `scripts/aggregate_eval_dataset.py` 检查 schema / 桶配比 / sizing（≥30 起步）/ 重复，并提醒去污染。
5. **版本化**：JSONL 进仓库随 git 版本化；打 `date_added`；事故用例永久留在回放桶。

⛔ 停止条件：拿不到真实生产样本只能纯手写/纯合成时，明确告知用户"评测集会偏易、可能测不出真实失败"，建议至少补对抗 + 失败回放。

## `/AI评测` 工作流

目标：搭评测体系并跑出可回归的评测报告。

### Step 1 · 定评测体系

- **维度+权重+阈值**（方法论 3.1）：选维度、定权重、定阈值；每条可判真假。
- **grader 分层**（方法论 3.2）：能用规则判的（格式/关键词/结构）用 deterministic grader；开放式质量（连贯/有用/忠实度）才上 LLM-judge。**确定性地板**：judge 前先用规则拦明显失败。
- **量规**（开放式才需要）：用 `assets/judge-rubric-template.md`，G-Eval 风格（先列 CoT 评估步骤再打分）、分维度、取证（引用原文）、含反啰嗦判据。

### Step 2 · judge 校准（用 LLM-judge 时必做）

- judge 模型**≠被测模型族**（防自偏好）。
- 钉死 judge 模型版本 + 量规版本（不可变合同）。
- 用 ≥50 条人工金标量 Cohen's κ（目标 0.6–0.8+）；不达标先改量规/换 judge，别拿不可信的 judge 出分。
- 成对比较做位置交换取平均；`temperature=0` 验稳定性。

### Step 3 · 跑自动化评测

```bash
python scripts/run_eval.py --dataset eval-dataset.jsonl --report eval-report.md --result eval-result.json
# 有基线时做回归 / A-B：
python scripts/run_eval.py --dataset eval-dataset.jsonl --baseline baseline-result.json --report eval-report.md
```

`run_eval.py`（纯标准库）输出：分桶/分维度/总通过率、pass@k、失败用例清单、与基线对比的回归项。LLM-judge 默认 dry-run（不强依赖外部模型），需要真打分时按 `references/eval-harness-guide.md` 接平台 API。

### Step 4 · 出评测报告

用 `assets/eval-report-template.md`：测试对象+版本、评测集（来源/桶/规模/版本）、评测体系（维度/量规/grader/judge 模型版本）、总分+分维度、pass@k、通过率、**失败用例 + 失败聚类**、与基线对比（回归项）、剩余风险。

## `/AI调优` 工作流（GEPA 式反思优化）

目标：基于评测报告做**定向**调优，并用回归证明没退步。

1. **失败归因**：从评测报告的失败用例聚类，提炼"为什么错"（ASI：错误信息、违反的判据、推理日志），不是只知道"错了"。
2. **回溯策略**：错误是策略缺失？还是提示词没覆盖？还是服务/字段问题？回到 `prompt-strategy-first.md` 的策略卡定位根因（一般化修，别只打补丁过拟合某条用例）。
3. **定向修改**：每轮只改 **3–5 处**，优先级：格式不合规 → 违反禁止项 → 核心流程缺失 → 细节体验；提示词版本化（`prompt-vX.Y.md`）。
4. **回归对照**：改后跑同一评测集，与上一版 A/B（`run_eval.py --baseline`）；**不回退已验证有效的改动**；某改动在 A 类升 B 类退时，保留候选在 Pareto 前沿分别评估。
5. **收敛**：覆盖率 ≥95%、格式合规 ≥95%、总达标 ≥75%、迭代 ≤5 轮。
6. **出调优报告**：用 `assets/tuning-report-template.md`，记录：问题归因、改了哪 3–5 处、A/B 对比数据、回归结论、采用/不采用理由、剩余风险。

> 重场景的自动化调优：把"分数 + 文本反馈"接给 GEPA/DSPy（`dspy.GEPA`）自动进化提示词（方法论 5.2）。本仓库默认人工驱动，重场景再映射。

## User Checkpoints（必须停下来问）

- 评测集只能纯手写/纯合成（拿不到生产样本）→ 告知风险并征求是否继续
- 没有人工金标可校准 judge → 是否接受 judge 未校准的分（或先只用规则 grader）
- judge 模型与被测模型同族（自偏好风险）→ 是否换族
- 调优要同时大改提示词 + 字段 + 服务 → 先冻结字段合同，分轮改

## Non-Negotiable Rules

- 不要在没有评测集/评测体系时就大改提示词（无标尺盲调）
- 不要把"judge 给了分"当成"分可信"（未校准的 judge 会骗你）
- 不要用规则能判的场景去烧 LLM-judge（确定性优先）
- 不要一轮改一大堆（隔离变量，每轮 3–5 处）
- 不要让事故用例从评测集里消失（永久留在失败回放桶）
