# 策略先行的提示词工程（Strategy-First Prompting）

> 一句话：**提示词不是"写"出来的，是从策略"翻译 + 验证"出来的。** 先把策略拆成可判定的规则，映射成提示词实现，再逐条核对覆盖率，最后用评测集迭代。
>
> 何时读：执行 `/AIPRD`、`/AI策划` 撰写 Prompt 合同前；或 `/AI调优` 回溯策略时。
> 配合：`policies/prompt-engineering-techniques.md`（22 技术选型）+ `policies/llm-eval-methodology.md`（评测）。

---

## 为什么"先定策略"

策略文档是给**人**读的（解释性、描述性："教师应灵活因材施教"）；提示词是给**模型**读的（指令性、可判定）。直接把策略原文塞进提示词 = 把模糊留给模型自由发挥 = 不可控。

```
❌ 复制策略："教师应灵活运用多种教学方法，注意因材施教"
✅ 翻译为指令："学生连续 2 次答错同一环节 → 降一个难度层级；首次即答对 → 跳过降级直接推进"
```

所以顺序固定是：**定策略（可判定规则集）→ 撰写（映射成提示词）→ 评测（覆盖率 + 评测集）→ 调优**。在没有"策略规则集"和"什么算好"之前就开写提示词，等于没有标尺地盲调。

## 六条设计原则

| # | 原则 | 要点 |
|---|---|---|
| **P1** | 翻译而非复制 | 每条策略翻成"模型可判定的指令"，不照抄原文 |
| **P2** | 精确 > 全面 | 一条可判定规则胜过十条模糊描述；过"可判定性测试"：能否回答"模型做到了吗？是/否" |
| **P3** | 位置决定权重 | 利用首因/近因效应：核心角色+硬约束放**开头**，禁止清单+输出格式+示范放**末尾** |
| **P4** | Token 是公共资源 | 删常识、合并同类、控总量（4000–6000 token 为最优区，超过后中段遵循度下降） |
| **P5** | 渐进式复杂度 | L1 直接指令 → L2 条件分支 → L3 思维链 → L4 少样本；只有测试证明不足才升级 |
| **P6** | 鲁棒性优先 | 每个决策点都给：正常路径 + 异常路径 + 兜底默认；无兜底=不可控 |

位置布局（P3）：

```
┌──────────────────┐ ← 最高关注（首因）
│ 角色 + 核心硬约束   │
├──────────────────┤
│ 主体流程 / 策略     │ ← 中等关注
├──────────────────┤
│ 严格禁止           │ ← 高关注（末尾）
│ 输出规范 + 示范     │ ← 最高关注（近因）
└──────────────────┘
```

## 五步转化流程（不跳步，每步出中间产物）

```
Step 1 解析策略 → 规则清单（编号 + 分类 + 硬/软约束）
Step 2 映射设计 → 每条规则 → 提示词实现方式（一对一）
Step 3 生成提示词 → 按位置布局组装 + 工程规范
Step 4 自检验证 → 逐条核覆盖率 + 可执行性
Step 5 测试迭代 → 跑评测集，按报告修正
```

### Step 1 · 解析策略

逐节扫描，把策略拆成**带编号**的规则（遗漏一条就是提示词缺陷）。按类别提取：

| 类别 | 提取什么 |
|---|---|
| 工作流程 | 阶段划分、步骤顺序、衔接条件 |
| 分类策略 | 按类型/场景的差异化处理 |
| 对象处理 | 不同用户/行为的应对规则 |
| 反馈/响应 | 各结果下的话术与行为 |
| 节奏/边界 | 轮次、长度、时间等量化限制 |
| 禁止行为 | 明确禁止的操作/话术 |
| 示范/案例 | 完整对话或案例（含隐含规则） |

要点：每条编号（F1/Q3/S5…）；区分**硬约束**（必须 100% 执行）与**软约束**（可按上下文调）；注意从**示范对话**和**表格每一行**里抽隐含规则。

### Step 2 · 映射设计

为每条规则定"提示词里怎么实现"（一对一，不允许"隐含覆盖"）：

| 规则类型 | 实现方式 | 形态 | 自由度 |
|---|---|---|---|
| 明确约束/禁止 | 直接指令 | "禁止 X，改用 Y" | 低 |
| 按条件处理 | 条件分支 | "若 A→B；若 C→D" | 中 |
| 多步骤操作 | 流程嵌入 | "步骤 1→2→3" | 中 |
| 需要模仿 | 少样本示范 | 示例对话/案例 | 低 |
| 量化约束 | 数值表格 | 指标+目标+上限 | 低 |
| 异常/边缘 | 决策框架 | 条件表/决策树 | 中 |

**自由度原则**：操作越脆弱（输出格式、安全边界）→ 指令越具体（低自由度）；越灵活（话术风格）→ 越宽松。冲突的两条规则 → 在提示词里**声明优先级**。

### Step 3 · 生成提示词（统一 Prompt 合同六段）

本仓库 Prompt 合同正文统一为六段（与 `SKILL.md`、`execution-sop.md` 一致）：

```
人设      ← 角色 + 核心硬约束（放开头，P3）
任务      ← 这个模块要产出什么
规则      ← 主体策略（条件分支/流程/量化表）
输出规范  ← JSON schema / 字段 / 格式 + 至少 1 个完整示例
严格禁止  ← 禁止清单（放末尾，P3），每条配替代方案
思考链    ← 需要推理时的内部判断步骤（CoT）
```

工程规范：结构化（分层标题做索引、条件用 if-then/表格而非段落）、精确性（消除"适当/灵活/注意"等模糊词、数字边界明确）、鲁棒性（异常兜底、防漂移、冲突声明优先级）、Token 效率（删常识/合并同类/控 4000–6000）。

少样本设计：从策略示范中精选 1–3 个，至少 1 个完整正常流程 + 1 个异常/降级；示范 ≤8 轮（保留首轮+转折轮+末轮）；**示范与规则矛盾时改示范，不改规则**。

### Step 4 · 自检验证（必做，不可跳）

**覆盖率检查**：拿 Step 1 规则清单逐条定位实现位置（section+行）、判断是否可执行、标 ✅已覆盖/❌缺失/⚠️部分。

```
覆盖率 = 已覆盖数 / 规则总数 × 100%   目标 ≥95%，低于则补充重查
```

**可执行性检查**：每条过 可判定 / 无歧义 / 不矛盾（或已声明优先级）/ 有边界 / 有兜底。

产物落到 `assets/prompt-strategy-card-template.md`（提示词策略卡）：策略→指令映射表 + 覆盖率 + 六段 Prompt。

### Step 5 · 测试迭代

进入 `/AI评测`：用评测集跑（见 `eval-driven-development.md`）；按评测报告**每轮只改 3–5 处**，修复优先级：格式不合规 → 违反禁止项 → 核心流程缺失 → 细节体验；版本化（`prompt-vX.Y.md`）；不回退已验证有效的改动。

达标标准：

| 指标 | 及格 | 良好 | 优秀 |
|---|---|---|---|
| 策略覆盖率 | ≥90% | ≥95% | 100% |
| 格式合规率 | ≥90% | ≥95% | 100% |
| 总达标率（有评测集时） | ≥60% | ≥75% | ≥85% |
| 迭代轮次 | ≤5 | ≤3 | ≤2 |

达"良好"即可交付。

## 与 AI PRD 的衔接

AI PRD 的提示词模块（`execution-sop.md` Step 4）每个模块落成"输入→输出"合同。**策略卡是写 Prompt 合同前的前置产物**：先有策略卡（规则集+映射+覆盖率），再把六段 Prompt 填进 PRD 的提示词模块表格。这样 PRD 里的每段 Prompt 都可回溯到策略原文，调优时能精准回溯。
