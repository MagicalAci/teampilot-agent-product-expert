# 执行 SOP

## 目标

围绕同一项 AI 能力或 AI 工作流，稳定产出：

1. AI PRD
2. AI脚本
3. AI测试报告
4. 如有需要，再补开发脚本压缩包

## 入口模式

### `/AI策划`

全流程：`AIPRD -> AI脚本 -> AI测试 -> 可选打包`

### `/AIPRD`

只做 AI PRD。

### `/AI脚本`

只做 AI 脚本、最小接入材料和开发调用文档。

### `/AI测试`

只做 AI 脚本测试并输出测试报告。

## 标准流程

### Step 1. 冻结任务卡

先确认：

- 本轮要交付什么
- 给谁看
- 是全流程、PRD、脚本，还是测试
- 是否有官方模板、固定章节顺序、平台配置、输出字段约束
- 当前是否已经有上游资产：PRD、脚本、测试日志、Prompt 文档

如果用户给了 Feishu / Wiki 模版，先读模版，再开始写文档。

### Step 2. 受限定义输入数据结构

这一章必须先写，不能跳。

要完成：

- 总数据结构
- 一级对象字段
- 数组项与嵌套对象字段
- 必填 / 选填
- 字段回退口径
- 旧协议兼容口径

这一章的目标是先把"AI 允许看到什么、脚本如何解释这些数据"锁死。

### Step 3. 撰写 AI 策略

在字段冻结后，才进入策略。

必须覆盖：

- 生成顺序
- 模型与平台配置
- 模块依赖关系
- 模块级筛选 / 排序 / 拼装逻辑
- 字段回退口径
- 结构化输出兼容策略
- 业务红线

如果策略里还在回头改字段，说明上一步没冻结好。

### Step 4. 搭建提示词模块（PRD 阶段）

每个模块都要落成一个"输入到输出"的合同。

最少包含：

- 输入参数
- 输出参数
- 输出示例
- 前端映射或下游映射
- prompt 详情

Prompt 正文必须用统一结构：

- 人设
- 任务
- 规则
- 输出规范
- 严格禁止
- 思考链

### Step 5. 脚本开发与测试

只有前 3 步稳定后，才进入脚本层。

脚本层至少包含：

- 核心生成器
- 模型客户端
- route / handler 示例
- demo 脚本
- 输入输出样例
- 单元测试
- 开发 README / 调用文档
- 脚本包清单

如果本轮是 `/AI脚本`，至少要先读取已冻结的 AI PRD 或字段合同，不能在字段还漂移时直接写代码。

`/AI脚本` 可以包含最小可运行自测 / 单测，但不默认产出正式 AI 测试报告；正式测试报告由 `/AI测试` 负责。

机械工作优先复用：

- `scripts/init_ai_planning_delivery.py`
- `scripts/validate_ai_planning_assets.py`
- `scripts/package_ai_script_bundle.py`
- `scripts/export_svg_to_png.js`

### Step 6. 形成 AI 测试报告

AI 测试报告不是日志归档，而是验证结论。

至少记录：

- 跑了哪些命令
- 哪些是真实调用
- 哪些是单元测试
- 结构化输出是否稳定
- 结果是否通过
- 还有什么限制

如果本轮是 `/AI测试`，默认产出就是这份报告，不要求顺带重写 PRD。

### Step 7. 打开发交付包

只带"开发最小接入"需要的内容。

包内必须有：

- 源码
- 测试
- 输入输出样例
- Prompt 文档
- AI PRD
- AI测试报告
- README / 调用文档
- 脚本包清单

## 停止条件

以下任一情况出现，必须暂停并回到用户：

- PRD 模版读不到
- 平台配置与当前实现不一致
- 输出字段又变了
- 真实调用失败
- 文档和代码出现口径冲突
