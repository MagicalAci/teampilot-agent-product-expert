# 豆包爱学 Benchmark

这是 `single-product-competitor-analysis` 当前的基准案例库。

它的作用不是证明“这个 skill 能生成一份 Markdown”，而是证明：

- 高标准成稿应该长什么样
- 章节图证如何进入正文
- 写作计划、体验报告、审核门禁、事实核查如何互相配合

## 建议查看顺序

1. `06-writing/doubao-aixue-overall-20260310-r2.md`
2. `06-writing/WRITING_PLAN.md`
3. `04-experience/EXPERIENCE_REPORT.md`
4. `05-synthesis/REVIEW_GATE.md`
5. `07-factcheck/`
6. `08-visuals/VISUAL_PLAN.md`

## 这个 benchmark 保留了什么

- 主报告正文
- 写作计划
- 体验报告
- 审核门禁
- 三轮事实核查结果
- 正文实际引用到的关键截图与图表
- 主报告直接回指到的关键平台总结

## 关于小红书评论层的说明

这份 benchmark 已按当前案例库标准做过一次归档整理。

当前 skill 的最新标准仍然是：

- 小红书不能只抓帖子层
- 必须继续深挖重点帖详情页
- 重点帖评论层默认要抓全

这份案例当前已经满足“成稿 benchmark”门禁，更适合用来看：

- 成稿结构
- 图证进入章节的方式
- 写作与审核流水线

同时也要知道：

- 这里保留的是支撑成稿判断的最小必要集，不是完整 raw 数据包
- 如果你要复刻完整采集过程，仍应按当前 skill 协议重新抓取评论层与原始数据

## 没有整包搬进来的内容

这里没有拷贝完整 raw 数据、全部抽帧和整套采集过程文件。

保留原则是：

- 优先保留“能完整读懂成稿”的最小必要集
- 优先保留“能反推 workflow”的关键过程文件
- 优先保留正文实际引用到的图片和证据文档

## 用这个案例时要看什么

重点不是抄句子，而是看这 4 件事：

1. 图证是否真的服务于章节判断
2. `WRITING_PLAN.md` 是否足够约束章节质量
3. `EXPERIENCE_REPORT.md` 的素材如何进入正文
4. `REVIEW_GATE` 和 `fact-check` 如何把质量卡住
