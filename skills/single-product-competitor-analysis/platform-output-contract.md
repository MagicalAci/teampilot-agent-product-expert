# 平台输出协议

每个平台级搜索子代理都必须输出两类文件：

- `summary.md`
- `data.csv`

## 1. 路径约定

```text
03-platforms/<platform>/summary.md
03-platforms/<platform>/data.csv
```

## 2. `summary.md` 负责什么

`summary.md` 不是“正式总报告”，但它必须是**平台分析文档**，而不是“采集过程汇报”。

优先写这些：

- 这个平台上用户在讨论什么
- 用户把产品当成什么工具 / 什么角色
- 哪些功能点最容易被提及、被推荐、被吐槽
- 哪些用户群最活跃
- 典型行为路径是什么
- 有哪些关键原话、关键帖子、关键链接值得进正文

不要让这些内容占据主体：

- 调用了多少轮工具
- 去噪做了几步
- 安装 / 登录 / 初始化过程
- “本轮如何采集”的操作叙事

这类内容最多只保留成很短的样本说明；详细过程记录应进入 `PROCESS_LOG.md` 或采集过程文档。

推荐固定为 6 个章节：

### 2.1 本轮任务概况

- 平台名称
- 当前这批样本大致覆盖了什么
- 当前样本量与阈值判断
- 这批数据最适合回答什么问题

### 2.2 样本与证据面

- 本轮抓取范围
- 代表性样本
- 当前证据边界

### 2.3 关键事实信号

- 高频功能点
- 高频用户类型
- 高频正向 / 负向反馈
- 关键帖子、关键原话、关键链接

### 2.4 平台洞察与启发

- 基于本平台证据的初步洞察
- 候选判断
- 对后续章节最有帮助的启发

### 2.5 风险、噪音与缺口

- 噪音与风险
- 还缺什么
- 建议补抓方向

### 2.6 需要用户确认 / 补充

- 是否接受当前样本量
- 是否有误读需要纠正
- 是否需要补充私有信息 / 登录后页面 / 录屏截图
- 是否需要优先深挖某类用户 / 某个问题

这一节的作用不是“客套追问”，而是：

- 补证据缺口
- 纠正 AI 对平台内容的误读
- 让用户决定是否接受当前样本量
- 请求用户提供公开渠道拿不到的材料
- 帮助下一轮采集聚焦更重要的问题

允许写：

- 基于本平台证据的候选判断
- 对产品、传播、SEO、商业化等后续章节有帮助的局部洞察
- 明确标注为“待后续核对”的策划启发
- 平台级的用户画像、行为路径、功能认知

不要写：

- 直接替代正式报告的最终结论
- 只靠单平台就下全局竞争判断
- 直接给最终版 `跟 / 避 / 绕 / 观察`
- 大段采集过程复盘，尤其是工具调用细节

## 3. `data.csv` 负责什么

`data.csv` 只负责沉淀结构化原始数据。

最少字段：

- `id`
- `source_id`
- `source_channel`
- `title`
- `url`
- `local_path`
- `tier`
- `platform`
- `collected_at`
- `text`

推荐补充字段：

- `author`
- `publish_date`
- `likes`
- `comments`
- `shares`
- `keyword`
- `rating`
- `tags`

### 3.1 小红书专项要求

`xiaohongshu` 平台不能只沉淀帖子层。

`summary.md` 额外必须写清：

- 本轮帖子池总量
- 已做 `detail` 深挖的帖子数量
- 评论总量
- 是否已经覆盖重点帖评论层
- 如果评论未抓全，缺口在哪里

`data.csv` 推荐补充：

- `record_type`：`post` / `comment`
- `parent_id`
- `note_id`
- `comment_level`
- `post_url`

红线：

- 只有帖子行、没有评论行时，不能把结论写成“小红书评论显示……”
- 没有评论层就默认不能算高标准放行，除非用户明确接受当前样本

## 4. 样本量规则

- 默认目标：每个平台至少 `200` 条/篇有效样本
- 如果低于 `200`：
  - AI 不能直接假装样本足够
  - 必须回到对话里问用户：
    - 是否接受当前样本量
    - 是否扩大关键词范围
    - 是否增加平台

## 5. 适用平台

- `web`
- `appstore`
- `xiaohongshu`
- `weibo`
- `bilibili`
- `zhihu`
- `seo`
- `pricing`
- `manual`
