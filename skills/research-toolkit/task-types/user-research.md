# 用户研究

## 适用场景

- 聚焦用户痛点、需求和行为的深度研究
- 产品已存在但需要理解"用户到底要什么"
- 社媒口碑是核心数据源，重采集深度
- 典型触发：`/用户调研 [产品名/领域]`

## 章节结构

| 序号 | 章节名 | 核心目标 | 主要证据来源 | 必备表格/图表 |
|------|--------|----------|-------------|--------------|
| 1 | 总结 | 用户核心洞察浓缩，最后回写 | 各章结论汇总 | 洞察摘要表 |
| 2 | 用户画像与分群 | 还原目标用户的人口统计、行为特征、分群 | 社媒帖子 + 评论 + 搜索 | 用户分群表、画像卡片 |
| 3 | 核心需求与场景 | 用户在什么场景下产生什么需求 | 社媒帖子 + 搜索 | 需求场景矩阵、JTBD 表 |
| 4 | 痛点深挖 | 三层证据体系：帖子层→本帖评论层→他帖评论穿透层 | 社媒帖子 + 评论（必须分层） | 痛点分级表、三层证据对照表 |
| 5 | 竞品满足度分析 | 现有产品对用户需求的满足程度 | 社媒评价 + App Store 评论 | 需求满足度矩阵（需求×竞品） |
| 6 | 需求优先级与机会 | 基于痛点强度和满足缺口排优先级 | 全篇证据交叉 | 需求优先级矩阵（影响×频率×缺口） |
| 7 | 行动建议 | 产品/运营/内容层面的可执行建议 | 各章结论 | 行动建议表（P0/P1/P2） |

## 写作顺序

按 2→3→4→5→6→7→1 推进。第 4 章（痛点深挖）是全篇核心，写作时间最长，必须严格区分三层证据。

## 三层证据体系

第 4 章强制使用三层证据结构：

| 层级 | 定义 | 采集方式 |
|------|------|----------|
| 帖子层 | 原始帖子正文中表达的痛点/需求 | 关键词搜索采集 |
| 本帖评论层 | 该帖子下评论区的补充、认同或反驳 | 深入帖子抓取评论 |
| 他帖评论穿透层 | 其他相关帖子评论中出现的同类痛点 | 扩展搜索 + 交叉验证 |

每个痛点结论必须标注证据来自哪一层，且至少有两层交叉确认。

### 痛点分类维度

分析时必须区分以下需求类型：

| 类型 | 定义 | 识别标志 |
|------|------|----------|
| 功能需求 | 用户需要某个具体功能或解决方案 | "希望有…""能不能…""要是能…" |
| 情绪需求 | 用户需要情感支持或认同 | "好焦虑""崩溃了""太难了" |
| 社交需求 | 用户需要社交认同或面子维护 | "别人都…""怎么跟人说…" |
| 信息需求 | 用户需要资料、教程、指导 | "怎么…""哪里能找到…""求推荐" |
| 代做需求 | 用户想让别人替自己完成 | "有没有人帮…""求代…""多少钱" |

不同类型的痛点对产品的启示完全不同，不能混为一谈。

## 采集要求

| 维度 | 目标 |
|------|------|
| 社媒采集 | 每平台有效样本 ≥ 300 条（本任务类型采样标准高于默认值） |
| 小红书 | 必须分别采集帖子正文和评论，不能合并计数 |
| 知乎 | 问答正文 + 回答 + 评论三层 |
| 评论穿透 | 高互动帖子（点赞/评论 top 20%）必须深入抓取评论 |
| 搜索引擎 | 需求词 + 问题词 + 痛点词，至少 60 条有效结果 |

## 门禁差异

| 门禁 | 要求 |
|------|------|
| 采集审核 | 审核代理需确认帖子层和评论层已分开归档 |
| 样本量检查 | 每平台 300 条，且评论层独立计数 |
| 事实核查 | 三轮：章节级 → 痛点交叉验证专项 → 全稿级 |
| 三层证据 | 第 4 章每个痛点必须标注证据层级和交叉确认状态 |
| 用户原话 | 关键结论必须附用户原话（含来源链接） |
