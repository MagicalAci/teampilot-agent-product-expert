# 采集与爬取手册

这份手册服务两种场景：

- `/竞品分析` 中的采集阶段
- 独立指令 `/爬取`

目标不是让脚本自动分析，而是：

- 用工具把资料抓回来、整理好、落成标准文件
- 让 AI 基于数据做平台总结、缺口判断和后续提问

## 0. 搜索必须拆子代理

搜索阶段不能由一个代理串行乱搜。

默认要求：

- 先列子代理清单
- 明确每个子代理负责什么
- 按批次并发执行
- 搜索结束后必须交给审核代理
- 不合格就打回，最多 5 轮

详细见：

- [subagent-search-protocol.md](subagent-search-protocol.md)
- [tooling-and-installation-protocol.md](tooling-and-installation-protocol.md)
- [review-gate-protocol.md](review-gate-protocol.md)

## 1. 核心原则

### 1.1 先核关键词，再启动采集

AI 必须先和用户核对：

- 核心词
- 别名词
- 功能词
- 长尾词
- 评价词
- 问题词
- 竞品词

### 1.2 平台产物必须标准化

每个搜索子代理都要按平台落：

- `summary.md`
- `data.csv`

默认平台：

- `web`
- `appstore`
- `xiaohongshu`
- `weibo`
- `bilibili`
- `zhihu`
- `seo`
- `pricing`

### 1.3 样本量不足要回问用户

- 默认每个平台至少 `200` 条/篇有效样本
- 低于阈值时，AI 必须先和用户确认下一步

### 1.4 平台总结由 AI 完成

脚本只负责采集、归档、计数和格式化。

平台总结应由 AI 来写，至少说明：

- 当前采回了什么
- 数据量够不够
- 有哪些明显信号
- 基于本平台证据形成了哪些初步洞察
- 哪些启发对后续章节最有帮助
- 哪些地方还不够
- 接下来该补什么

推荐章节顺序：

1. 本轮任务概况
2. 样本与证据面
3. 关键事实信号
4. 平台洞察与启发
5. 风险、噪音与缺口
6. 需要用户确认 / 补充

“需要用户确认 / 补充”的作用：

- 让用户决定是否接受当前样本量
- 纠正 AI 对平台内容或用户群的误读
- 请求补充公开渠道拿不到的素材
- 决定下一轮采集优先深挖什么

边界：

- 可以写平台级洞察和候选判断
- 不要把单平台总结直接写成正式总报告结论

## 2. 默认后端顺序

`/爬取` 默认优先使用现有可用能力：

1. MCP：Firecrawl
2. MCP：AppInsight
3. MCP：小红书 / B站 / 微博
4. 浏览器工具 + WebSearch / WebFetch
5. 可选 Shell 适配位：本地 crawler

说明：

- `~/crawlers` 相关方案目前视为“可选适配位”
- 不把本地 crawler 当默认前提
- 子代理必须先检查工具状态，缺安装就安装，缺登录就初始化
- 平台要求和最低阈值以 `platform-output-contract.md` 与 `tooling-and-installation-protocol.md` 为准

## 3. 采集范围清单

### 3.1 官网与产品页

至少覆盖：

- 首页
- 功能页
- 帮助中心
- 更新日志或关于页

### 3.2 评论与用户反馈

至少覆盖：

- 应用商店评论
- 社区问答
- 公开测评中的用户引用

### 3.3 社媒

优先覆盖：

- 小红书
- B站
- 微博
- 论坛或问答站

补充要求：

- 小红书不能只抓搜索命中的帖子列表
- 必须继续深挖重点帖详情页，把帖子下评论层一并抓回
- 如果后续要写“小红书评论区真实反馈”，必须能回指到评论层数据

### 3.4 SEO 与内容

至少覆盖：

- 博客列表页
- 落地页
- 品牌词 / 产品词 / 问题词承接页

### 3.5 商业化

至少覆盖：

- 定价页
- 会员页
- 试用页
- 权益说明页

### 3.6 体验素材

采集阶段要同步标记后续可能需要补的：

- 首页截图
- 核心功能页截图
- 会员页截图
- 登录后录屏

## 4. 平台标准产物

路径：

```text
03-platforms/<platform>/summary.md
03-platforms/<platform>/data.csv
```

含义：

- `summary.md`：AI 的单平台总结、缺口、下一步建议
- `data.csv`：结构化原始数据，不承载战略结论

详细字段见 [platform-output-contract.md](platform-output-contract.md)。

## 5. 什么时候必须请求用户补料

AI 发现以下情况时，必须回来问用户：

### 5.1 深层体验拿不到

- 原生 App 才有核心价值
- 登录后页面才有关键功能
- 会员后才有关键路径

### 5.2 商业化路径看不全

- 定价页不公开
- 订阅权益隐藏
- 支付前页需要登录

### 5.3 用户反馈样本偏弱

- 公开评论太少
- 平台噪音很大
- 样本严重偏向某一类用户

## 6. 一句话标准

好的采集不是“边写边搜”，而是：

`先把资料做深做全、按平台沉淀成标准文件，再让 AI 基于证据去提问、归纳和组织后续分析。`
